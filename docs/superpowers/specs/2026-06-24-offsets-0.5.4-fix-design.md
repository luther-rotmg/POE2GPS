# PoE2 0.5.4 Offset Hotfix — Design

**Date:** 2026-06-24
**Status:** Design approved. Ships as **v0.5.1** hotfix off `main` (branch `fix/offsets-0.5.4`).
**Scope:** Mechanical offset re-validation. No new behavior, no new reads, no UI.

## Problem

PoE2 patched to **0.5.4**. GGG inserted **24 bytes (0x18)** into the `AreaInstance`
struct, shifting every field at offset **≥ 0x580** by **+0x18**. The overlay's core read
chain (local player, entity maps, terrain, inventory gateway) now resolves to the wrong
addresses, so the radar reads garbage / nothing on 0.5.4.

## Evidence

User supplied four confirmed shifts, cross-checked against the GameHelper2 0.5.4
`GameOffsets` dump — all exactly **+0x18**:

| Field | Old | New |
|---|---|---|
| `ServerDataPtr` (GH2 "PlayerInfo") | 0x580 | 0x598 |
| `LocalPlayer` | 0x5A0 | 0x5B8 |
| `AwakeEntities` | 0x6C0 | 0x6D8 |
| `TerrainMetadata` | 0x8A0 | 0x8B8 |

A uniform +0x18 across four independent fields is the signature of a **single in-struct
insertion** between offset `0x11C` and `0x580`. That model fully determines the rest of
the change set.

## What is NOT changing (and why we're confident)

The +0x18 insertion is **contained to `AreaInstance`**. The following live in separate
structs and are therefore unaffected by an `AreaInstance`-internal insertion:

- `InGameState` hops: `AreaInstance @ +0x290`, `UiRoot @ +0x2F0`, `Camera @ +0x368`.
- `AreaInstance` **low** fields below the insertion: `AreaInfoPtr 0x0A0`,
  `CurrentAreaLevel 0x0C4`, `CurrentAreaHash 0x11C`.
- Entity + component offsets: `Render +0x138`, `Life` vitals, `Rarity +0x144`,
  `Positioned.Reaction +0x1E0`.
- `ServerData` internals + the inventory chain (`+0x48`, `+0x320`, …).
- `UiElement`, atlas, and item-mod offsets.

These are *almost certainly* stable. But **components have drifted independently before**
(the Life vital blocks slid in a prior patch), so we do not assume — we validate them live.
GameHelper2 is **not** a reliable oracle for our PoE2 component offsets (our own table
records GH2's `0xD20` / `0xBC` / `0xFC` as drifted); the live probes are the only real check.

## Validation — live, in-game on 0.5.4 (manual, run by Ryan)

Rebuild the Research project from this branch (so the probes use the corrected constants),
launch PoE2 0.5.4, get fully in-game (in a zone with monsters, not the menu), then run each
probe. Each line below states the PASS criterion.

| Probe | Confirms | PASS looks like |
|---|---|---|
| `--chain` | Whole chain + `LocalPlayer` resolves; catches `0x290`/`0x2F0` if they *also* moved | `LocalPlayerPtr … MATCH`; non-zero AreaInstance/InGameState |
| `--info` | The **low** AreaInstance fields didn't shift | Area code (e.g. `G1_town`), sane AreaLevel, non-zero AreaHash |
| `--vitals` | `Life` component (the one struct that drifted independently last patch) | Health/Mana/ES match the in-game orb values |
| `--rarity` | `AwakeEntities` walk + entity `Rarity`/`Reaction` | Sensible entity count; rarities/hostility look right |
| `--inventory --itemmods` | `ServerData → inventory` chain + mod ids | Inventories list with box dims; item mods render |

**All PASS → ship.** **Any FAIL → real additional drift**: report the probe + its output,
fix that specific offset, rebuild, re-run only the failing probe.

## Compliance

Read-only end to end: integer-constant edits + comment edits only. No new memory writes, no
input/automation, no pricing/market data. `scripts/compliance-gate.ps1` and
`scripts/scrub-strings.ps1 -SelfTest` must pass; `dotnet build POE2Radar.slnx -c Release`
must be clean.

## Release
