# v0.8.0 — Performance & Footprint Pass — Design

**Date:** 2026-06-27
**Branch:** `feat/v0.8.0-perf`
**Status:** Approved design — ready for implementation plan
**Audit source:** `docs/audit-2026-06-27-perf.md` (six-auditor head-to-toe sweep)

## Goal

Ship the **SOLO** (behavior-preserving, offline-buildable) optimizations from the 2026-06-27 audit so
POE2GPS leaves the lightest possible resource footprint — lower idle RSS, fewer per-frame
allocations, less render/world CPU — with **zero change to what memory is read** and **zero new
surface**. The larger raw-syscall wins that rewrite the *read pattern* are deferred to the in-game
validation session (see audit Bucket 2); they are explicitly **out of scope** here.

## Non-goals (deferred to the in-game session)

`ResolveComponent` bulk-read, `ReadReaction` value-cache, `TryReadMapElement` bulk-read,
`ReadChestOpened` cache, `PlayerVitals` world-thread split, `ReadMods` bulk-read, `Entities()`
list-reuse (the publishing-lifetime variant), terrain raw-buffer `ArrayPool`. Each changes a memory
read and needs a live entity/map view to confirm correctness — not solo-verifiable.

## Guiding invariant

Every change in this pass is **observably identical** to today's behavior — it removes redundant
work (allocations, repeated reads of stable values, duplicate computation), never alters output.
That is what makes the whole pass solo-safe: the existing test suite + build + compliance gates +
final review are a sufficient net, and the eventual in-game smoke is a confirmation, not a gate.

## Scope — themed task groups

Grouped by file/responsibility so each task is coherent and independently reviewable. Tasks that
touch `RadarApp.cs` run sequentially (SDD dispatches one implementer at a time — no parallel edits).

### Group A — Idle RSS: lazy mod-translation tables (Core)

`ItemModTranslator.Shared` (`ItemModTranslator.cs:36`) and `ModRanges.Shared` eager-load three
embedded tables (~5.3 MB raw → ~15–20 MB managed) at first class use, even though they're consumed
only by the **gear scorer (default OFF)** and rare lazy POI name resolves.

- Convert both `Shared` singletons to `Lazy<T>` (or equivalent deferred init) so the object graphs
  are not built until first actual use. Idle RSS drops ~15–20 MB.
- Audit `StatIdsForRenderedLine` / `_renderedIndex` (`:45,51`): confirm no **live overlay** path calls
  it (it's Research/offline-generator only). If confirmed, annotate as Research-only. `_renderedIndex`
  is already lazy — leave it.
- A change to a `static` getter's laziness is awkward to unit-test directly; where a focused Core
  test adds real signal (e.g. the translator still returns correct lines once forced to load), add it.
  Otherwise rely on build + existing suite.

### Group B — Unbounded session accumulators (Overlay/Web)

`SeenPoiLog` (`Web/SeenPoiLog.cs`), `EntityAtlasLog` (`Web/EntityAtlasLog.cs`), `ModCatalog`
(`Web/ModCatalog.cs`) grow monotonically with no cap.

- Add a configurable max-entry cap (defaults: SeenPoiLog 5 000, EntityAtlasLog 10 000, ModCatalog
  2 000): once at cap, stop inserting new keys (existing entries still update). One-way; no eviction
  churn.
- `ModCatalog`: switch the hot `SortedSet<string>` to `HashSet<string>`; sort only at `Save()`.
- These are pure-logic — **add focused unit tests** for the cap behavior (insert past cap → count
  frozen, existing keys still updatable; ModCatalog Save output still ordinal-sorted).

### Group C — Render-frame allocations in `RadarApp.Tick` (Overlay)

All fire on the render thread at FpsCap (up to 144+ Hz):

- `AtlasProjection()` `double[8]`/frame (`:1595-1600`) → fill a reusable `private readonly double[8]`
  field (or a small readonly struct) in place; safe because the caller reads it synchronously in the
  same `Tick()`.
- Atlas-route point lists (`:983-996`) → reuse render-thread scratch fields, same idiom as the
  existing `_atlasMarkFrame` / `_hpFrame`.
- `Probe()` `new List<nint>(13)` (`Poe2Live.cs:156`, hit at render rate via `TryResolve`) → reusable
  per-instance field cleared on entry (each `Poe2Live` is single-thread-owned; no lock).
- `ActiveCycleList()` per-cycle alloc (`:1655-1662`) → publish a `_defaultCycleTargets`
  (`volatile IReadOnlyList<RankedTarget>`) from the world tick; `ActiveCycleList` returns it directly.

### Group D — Render-frame allocations in `OverlayRenderer` (Overlay)

- `DrawSessionHud` per-frame `List<(string,bool)>` + interpolated strings (`:591`) → cache the
  formatted lines; rebuild only when the seconds-resolution session values change.
- `ParseColor` per entity + per landmark per frame (`:826,847`) → cache the parsed color once when a
  rule is loaded/edited. **Constraint:** `DisplayRule` lives in a layer that must stay free of the
  Vortice dependency — cache as a packed `uint`/`Color4` **on the renderer side** (memo keyed by rule),
  or as a Vortice-free packed `uint` on the rule, mirroring the HP-bar `PackColor` pattern. Do **not**
  introduce a Vortice type into Core.
- `DrawMonolithPanel` `OrderByDescending+Take+ToList`/frame (`:552`) → sort + cap once at publish
  time on the world thread; store the capped list in `MonolithRender`; renderer iterates plainly.

### Group E — World-tick CPU + threading (Overlay + Core)

- Compute `_campaign.Rank(...)` **once per tick** and pass it into `BuildRankedTargets` /
  `DirectorReconcile` / `CampaignReconcile` (`:1271-1301`) instead of 2–3× recompute.
- `BuildNavTargets` (`:1635-1647`): replace the `.Where/.Select/.Where/.OrderBy` chain with a single
  `foreach`; memoize `_displayRules.Resolve(e)` into a per-tick `Dictionary<nint,DisplayRule?>` shared
  with `BuildHpSpecs`; drop the `OrderBy` (F6 nearest-scan already re-sorts). Result list identical.
- Cache AreaHash / AreaLevel per `areaInstance` (`:1147-1148`); re-read only on instance change.
- Throttle `DirectorReconcile` to ~4 Hz (`:1292`), forced on zone change.
- `UpdateAtlas` median (`:2263-2264`) → reuse a `List<float>` field + in-place `Sort`.
- `ScanLootLabels` BFS (`Poe2Live.cs:1282-1312`) → hoist `Queue`/`HashSet` to reusable fields.
- `std::map` walk cap (`Poe2Live.cs:425`) → bound to `size*4 + 1024`. **Add a Core test** for the
  bound if the walk is reachable in isolation; otherwise verify by reasoning + review.
- `SnapshotSelection` double-lock/tick (`:1301,2030`) → `MaintainRoutes` returns the snapshot it
  already built; `WorldTick` reuses it.

### Group F — Data-race fix + low/hygiene (Overlay + Core)

- **Correctness:** `Tick()` (render thread) calls `_live.PlayerVitals` — the world reader stack — at
  `:937`. Switch to `_liveRender`. The world thread already triggers the offset self-heal at `:1157`;
  identical observable behavior, removes the latent cache-corruption race.
- **Correctness:** vital-offset re-latch guard (`Poe2Live.cs:336`) — clear `_vitalOffsetsResolved`
  when a latched read yields `HpMax==0 || HpCur>HpMax`, with a small backoff so a legit 0-HP login
  frame doesn't thrash. **Add a Core test** for the re-latch trigger.
- `Landmark.Key` interpolation per access (`Poe2Live.cs:138`) → compute once, store on the record.
- `ReconcileTrackers` LINQ (`:1971`) → manual two-pass with a reused scratch list.
- `ToggleSelectionCore` `string.Join` inside `_navLock` (`:1945`) → build the log line outside the
  lock (or log only the delta).
- `DiscoverMapElements` BFS per-area alloc (`Poe2Live.cs:1028-1055`) + atlas chunk 1 MB buffers
  (`Poe2Atlas.cs:133,200`) → hoist to fields (consistency with `_entQueue`/`_entVisited`).

### Group G — Integration sweep + final whole-branch review

Full build (0W/0E), full xUnit suite, `compliance-gate.ps1`, `scrub-strings.ps1 -SelfTest`, then the
final reviewer over the whole branch diff.

## Architecture / boundaries

No new subsystem and no new public API. Each change is local to the method/field it names. The only
cross-file plumbing introduced:
- Group C: a new published `_defaultCycleTargets` list (world→render, volatile-swap idiom already used
  for `_navTargets`/`_rankedTargets`).
- Group D: `MonolithRender` carries an already-sorted/capped list; the per-tick Resolve memo dict is a
  world-thread-local field shared by `BuildNavTargets` + `BuildHpSpecs`.
- Group E: `_campaign.Rank(...)` result threaded as a parameter into the three reconcile consumers.

All caches added are single-thread-owned (world or render) or volatile-published — no new locks.

## Testing

The test project references **Core only**, so:
- **Unit-tested (Core):** cache-cap behavior (Group B), the std::map walk-cap bound and vital re-latch
  guard (Groups E/F) where reachable, and any lazy-load correctness assertion that adds signal
  (Group A).
- **Build + reasoning + review (Overlay):** the render/world-thread refactors (Groups C/D/E/F in
  `RadarApp`/`OverlayRenderer`) have no Overlay test harness; they are behavior-preserving by
  construction. Net: `dotnet build` 0W/0E (TreatWarningsAsErrors), full suite green, both compliance
  gates green, final review.
- **Deferred confirmation:** the next in-game session smoke-checks that the radar/atlas/HUD look
  identical (no perf change altered output).

## Compliance

Pure reads / allocation hygiene / CPU reduction. No new game-process interaction, no input, no memory
writes, no pricing. The data-race fix makes existing reads *more* correct. Compliance gate and
string-scrub stay green unchanged. README badge stays `supports PoE2 0.5.4` (no offset changes).

## Out of scope (YAGNI)

- All Bucket-2 in-game items (listed under Non-goals).
- The `Entities()` list-reuse publishing variant — only the trivial capacity-hint is even considerable
  and it's marginal; skip both here to avoid a publishing-lifetime change.
- Any allocation pooling beyond reusing single-owner scratch fields (no `ArrayPool` in this pass —
  that's the deferred terrain item).
- Audio Alerts + Community Presets — those are **v0.9.0** (separate spec/plan/SDD).
