## 1. Goal

Give players a **filter engine that highlights items matching desired affix combinations** across every item surface the tool touches (ground drops today; equipped items, inventory, and stash tabs across two release drops), plus two long-requested `/map` view improvements: **larger fog reveal radius** and **mousewheel + keyboard zoom**.

All three requests came from the same community feedback thread:
1. `/map` reveal radius is too small — "*I have to get within an inch of the wall for the line to show up*"
2. Add zoom to `/map` (mousewheel + `+`/`-` keys)
3. Highlight items in inventory/character/stash/ground that match user-defined affix combos ("*maximum Energy shield AND faster recharge*"), replacing "*abysmal default item tooltips*" as the mental model

## 2. Ship structure — one spec, two drops

**v0.31 (this drop) — the accessible 90%:**
- `/map` reveal radius fix + zoom
- Item Filter engine (DSL, storage, dashboard editor)
- Highlight on **ground items** (renderer already ships colored borders — trivial extension)
- Highlight on **character equipment slots** (data already flows via God-Roll Detector; only needs UiRoot probe + screen projection)
- Preset library (5–8 starter filters shipped disabled)

**v0.32 (follow-up drop, same spec) — the fragile 10%:**
- Highlight on **player inventory panel** (needs UI-panel walker; server-side reads already ship)
- Highlight on **stash grid tabs** (regular / quad / jewel / map / relic — the InventoryStruct layout is reused, per scout)

**Deferred to v0.33+ / individual beads:**
- Stash **specialty tabs** (currency / fragment / essence / delirium / expedition) — each is a bespoke UI class with its own offset probe; treat each as a follow-up.
- Replacement clean-tooltip on hover (user's second wish; explicitly out of scope per LO 2026-07-12 — "*just highlight*").

## 3. Global Constraints (bind every task)

- **Read-only**: never write to game memory, never inject input. All data via `MemoryReader` reads only. This is the tool's non-negotiable compliance invariant — the compliance gate at `scripts/compliance-gate.ps1` enforces it.
- **No feature gates on functional behavior**: the filter engine is a cosmetic-highlight system. It never hides, blocks, or replaces game UI. Existing item tooltips continue to render exactly as the game draws them.
- **Zero performance impact when idle**: filter matching runs only when the affix reads happen anyway (already ~30 tick cadence on equipped items and once-per-drop on ground items). Dashboard-side match-counter runs on the SSE snapshot.
- **String scrubber must pass**: new strings (English filter names, presets) go through `scripts/scrub-strings.ps1 -SelfTest` with no internal-tooling leaks.
- **Attribution sentinel must pass**: no reference to `superpowers/`, `SDD`, `Sikaka`, or `GameHelper` in any user-visible string, changelog entry, in-app text, or release notes.
- **Backwards-compat**: an existing user with no `item_filters.json` sees zero behavior change until they enable a filter. The presets ship `enabled: false` on first-run copy.

## 5. `/map` reveal radius fix

### Root cause (from scout `map-radius`)

The `/map` view renders terrain lines only inside a hardcoded 24-terrain-cell disc around the player (fog canvas + `destination-out` arc). Terrain outside the disc is masked entirely by opaque fog. In actual play — especially inside atlas nodes and large open zones — 24 cells leaves the reveal ring too tight to see approaching walls, mechanics, and mob spawns before you're already in them.

### Design

**Client:**
- Bump `REVEAL_RADIUS_CELLS = 24` → `60` at [src/POE2Radar.Overlay/Web/Assets/map.js:10](src/POE2Radar.Overlay/Web/Assets/map.js#L10).
  - 60 = 2.5× current (in-range of user's "double or triple"), aligned with the existing `AudioAlertRadiusCells = 60` so proximity systems agree.
- Read `revealRadiusCells` from `/api/settings` at startup; if present, override the constant. If the setting is absent or malformed, keep 60 as the default.
- Debounce settings-fetch overrides (they hit at SSE reconnect cadence anyway).

**Server:**
- Add `WebMapRevealRadiusCells` int to `RadarSettings` (default `60`).
- Surface via `/api/settings` GET (JSON field `webMapRevealRadiusCells`) and POST (round-trip to `settings.json`).
- Round-trip test: setting persists across dashboard reloads and app restarts.

**Dashboard row:**
- Add a numeric input to Settings → Advanced → "Web map reveal radius (cells)" with range 20–200, step 10, tooltip explaining that higher = more terrain visible at once.

### Non-goals

- No animated reveal (fog stays a hard disc; no gradient / feathering — the game itself doesn't gradient-fade discovered terrain and matching game behavior is simpler + faster).
- No per-zone override (single global setting — YAGNI).

## 6. `/map` zoom

### Root cause (from scout `map-zoom`)

`state.zoom` is loaded from `localStorage` once at `map.js:23` and **never mutated after load**. Terrain uses `ctx.setTransform` with `state.zoom` as the scale factor; all other layers (entities, monoliths, landmarks, atlas, paths, player arrow) use `worldToScreen(dx, dy, state.zoom)` for manual projection. There is a single keydown handler at `map.js:702` that only toggles GPS mode; no wheel handler exists.

### Design

**State additions:**
- `state.zoom` becomes a `float` (was integer) with `parseFloat` on load; clamp to `[0.5, 32]`.
- Add `state.panX = 0`, `state.panY = 0` — supports **cursor-anchored zoom** (offset stays aligned with the mouse when scaling up/down).
- Refactor the six `cx = cw/2, cy = ch/2` sites (drawEntities, drawMonoliths, drawLandmarks, drawAtlas, drawPaths, drawPlayer) + `applyIsoTransform`'s tx/ty to `cw/2 + state.panX`, `ch/2 + state.panY`.

**Mousewheel handler:**
```javascript
state.canvas.addEventListener('wheel', (e) => {
  e.preventDefault();
  const factor = e.deltaY < 0 ? 1.15 : 1 / 1.15;
  const newZoom = Math.max(0.5, Math.min(32, state.zoom * factor));
  const scaleRatio = newZoom / state.zoom;
  // Cursor-anchored pan correction so the pixel under the cursor stays put.
  const cw = state.canvas.width  / state.dpr;
  const ch = state.canvas.height / state.dpr;
  state.panX = e.clientX - (cw / 2 + state.panX) * scaleRatio + state.panX * scaleRatio - (e.clientX - cw / 2);
  state.panY = e.clientY - (ch / 2 + state.panY) * scaleRatio + state.panY * scaleRatio - (e.clientY - ch / 2);
  state.zoom = newZoom;
  localStorage.setItem('mapZoom', newZoom.toFixed(2));
  // Fog canvas doesn't need recompute — it re-renders every frame anyway.
}, { passive: false });
```
(Cursor math simplifies to `state.panX += (e.clientX - cw/2 - state.panX) * (1 - scaleRatio)` — spec-normal form.)

**Keyboard handlers** (extend `map.js:702` block):
- `+`, `=` → zoom in by `1.15` factor, viewport-center anchored (no pan correction, or equivalently: mouse at center)
- `-`, `_` → zoom out by `1/1.15`, viewport-center anchored
- Both keys share the mousewheel's clamp + localStorage persistence
- Guarded by existing `!e.repeat && !input/textarea/contenteditable` check

**HUD indicator:**
- Update the existing top-left readout at `map.js:631` to include `z${state.zoom.toFixed(1)}` (fits alongside the current fps counter).

### Non-goals

- No pan drag (arrow keys, click-drag) — player recentering is automatic on movement; explicit pan not requested.
- No touch/pinch — desktop-only tool; no touch surface.
- No zoom presets / +/- buttons in the HUD — keyboard + wheel cover the ask.

## 7. Item Filter engine

### Storage — `config/item_filters.json`

Persistent list of user filters. Schema:
```jsonc
{
  "filters": [
    {
      "id": "es-jeweler",
      "name": "ES Jeweler",
      "enabled": true,
      "color": "#FF8800",       // hex, drawn as border color
      "priority": 100,           // higher wins on multi-match; default 100
      "requirements": [          // AND across all — every requirement must pass
        {
          "statId": "local_energy_shield_pct",
          "op": ">=",
          "value": 10,
          "scope": ["prefix", "suffix"],  // optional; empty = both
          "maxTier": null                  // optional; null = any tier
        },
        {
          "statId": "faster_start_of_es_recharge_pct",
          "op": ">=",
          "value": 5
        }
      ]
    }
  ]
}
```

**Field semantics:**

| Field | Type | Semantics |
|---|---|---|
| `id` | string | Stable slug (auto from name; user can override). |
| `name` | string | Display label in dashboard + tooltip on hover. |
| `enabled` | bool | Master toggle. Disabled filters never match. |
| `color` | hex string | Border color for highlighted items. `#RRGGBB` or `#RRGGBBAA`. |
| `priority` | int | Higher wins on multi-filter matches. Default 100. |
| `requirements` | array | AND-list — every requirement must match. Empty = never matches (guardrail). |
| `requirements[].statId` | string | RePoE stat id (from `poe2_stat_descriptions.json` / `poe2_mod_ranges.json`). Stored as stable ground truth. |
| `requirements[].op` | enum | `>=`, `<=`, `==`, `between`. `between` uses `value` + optional `valueMax`. |
| `requirements[].value` | number | Rolled-value threshold. If `op == ">= 0"`, effectively presence-only. |
| `requirements[].valueMax` | number? | Only used when `op == "between"`. |
| `requirements[].scope` | string[]? | `["prefix"]`, `["suffix"]`, `["prefix", "suffix"]` — restricts which mod slot the stat can be on. Empty/absent = any. |
| `requirements[].maxTier` | int? | Only match if the mod's tier (from `ModRanges.tier`) is ≤ this. |

**Ordering:** filters are stored in list order. On multi-match, sort by `priority DESC`, then by list order.

### Match semantics — `ItemFilterEngine`

New class `POE2Radar.Core.Game.ItemFilterEngine`:
```csharp
public sealed class ItemFilterEngine {
    public ItemFilterEngine(string filePath, ModRanges modRanges, ItemModTranslator translator);
    public IReadOnlyList<FilterRule> All { get; }
    public int Generation { get; }             // mutation counter; renderer clears memoization

    /// <summary>Returns all filters that match the given item's affixes, sorted by priority DESC.
    /// Empty list = no match. First entry = winning filter (its color is used for the border).</summary>
    public IReadOnlyList<FilterRule> Match(IReadOnlyList<RawAffix> implicit, IReadOnlyList<RawAffix> explicit);

    /// <summary>Match against a Poe2Live.InventoryItem — routes affixes and applies scope/tier constraints.</summary>
    public IReadOnlyList<FilterRule> Match(Poe2Live.InventoryItem item);

    public void Replace(IEnumerable<FilterRule> rules);   // dashboard save
    public void Add(FilterRule rule);
    public void RemoveAt(int index);
    public void Update(int index, FilterRule rule);
    public void Move(int fromIndex, int toIndex);
}
```

**Match algorithm** (per-item):
1. For each filter in `_filters`:
   1. If `!filter.Enabled`, skip.
   2. For each requirement:
      1. Resolve `requirement.statId` → the set of affix values on this item that carry that stat (via `ItemModTranslator.StatIdsFor(modId)` reverse lookup, cached).
      2. If `scope` set, restrict to affixes whose mod-id prefix matches (`Poe2Live.RawAffix.ModId` → `ItemModTranslator.SlotOf(modId)` gives `Prefix`/`Suffix`/`Implicit`).
      3. If `maxTier` set, restrict to affixes with `ModRanges.TierFor(modId) <= maxTier`.
      4. Apply `op` to each candidate value; if ANY passes, this requirement is satisfied.
      5. Otherwise, this requirement fails → this filter fails, break.
   3. If all requirements satisfied, add filter to matched list.
2. Sort matched by `Priority DESC, ListIndex ASC`; return.

**Memoization:** engine caches per-item-address matches on a `Dictionary<nint, (int gen, IReadOnlyList<FilterRule>)>`. Invalidated when `Generation` bumps.

### Presets — `default_item_filters.json` (embedded resource)

Ship 5–8 curated starter filters — disabled by default:
- ES Jewel (max ES ≥ 10% + ES recharge ≥ 5%)
- Life Chest Baseline (max life ≥ 60 + any resist ≥ 20%)
- Rare Amulet Baseline (any tri-resist requirement)
- Cast Speed Wand (cast speed + spell damage)
- Resist Ring (dual-resist ≥ 25% each)
- Faster-attacks Weapon (attack speed prefix)
- ES/Life Chest (both max ES ≥ 8% and max life ≥ 40)
- Movement Speed Boots (any tier)

**First-run:** if `config/item_filters.json` does not exist, copy the shipped presets in with `enabled: false` and log `"Item filters: seeded 8 starter presets (disabled by default)."` to console. Any subsequent user edit replaces this file; the presets can be re-loaded via a dashboard button `"Restore starter presets"` (appends only, never removes user edits).

### Non-goals

- OR-groups nested inside a single filter (**deferred to v0.33+** — LO chose multi-filter-as-OR semantic, per brainstorm Q3).
- Filter tagging (`Tag: life`, `Tag: offense` — deferred).
- Regex on rendered lines (statId is stable; skip regex).
- Community filter sharing / import-export (deferred to a v0.32+ card).

**One-shot Research session** (~30 min, LO or ENI at desk):

- Snapshot `UiRoot.Children` — record the persistent set of child indices whose `visible-bit` is currently 0.

If the child index is unstable across game restarts (unlikely — atlas one has been stable), fall back to a signature-text walk (BFS UiRoot for a "Character" header string, walk up to the panel root — same pattern Ritual uses).

### Data path (already flows)

- `Poe2Live.ReadInventory` already returns per-slot `InventoryItem { Item ptr, InventoryId, Affixes }` for every equipped item, on world thread ~30 tick cadence. This is the God-Roll Detector's data source.
- **Zero new memory-read code required for the data layer.**

### New: per-slot screen rect

**New class** `POE2Radar.Core.Game.Poe2Character`:
```csharp
public sealed class Poe2Character {
    /// <summary>Locates the CharacterPanel UiElement + walks its slot children by fingerprint,
    /// returning the visible screen rect for each equipment slot. Returns empty when the panel
    /// is closed / not visible. Self-healing to layout drift via signature backtracking.</summary>
    public IReadOnlyDictionary<EquipmentSlot, RawRectF> ReadEquipmentSlotRects(nint uiRoot, int winW, int winH);
}

public enum EquipmentSlot {
    Weapon1, Offhand1, Weapon2, Offhand2,
    Chest, Helmet, Gloves, Boots, Belt,
    Amulet, Ring1, Ring2,
    Flask1, Flask2, Flask3, Flask4, Flask5   // optional row; v0.32 if not v0.31
}
```

**Walk pattern:**
1. `Poe2Live.ResolvePanel(uiRoot, CharacterPanel.UiRootChildIndex)` → get panel UiElement.
2. If not visible (`Flags & 0x0B == 0`), return empty dictionary.
3. Walk panel's children by known layout order (top row = helmet + amulet, upper-mid = weapon1 + chest + weapon2, etc.). Fingerprint each slot child by its relative position + size (analogous to Runeforge's flag-fingerprint pattern).
4. For each fingerprinted slot, call existing `TryUiElementRect(child, winW, winH)` (Poe2Live.cs:1251) → returns the projected screen rect.
5. Return dictionary keyed by `EquipmentSlot` enum → `RawRectF`.

**Bake the slot layout order** (from Research probe) into a static `EquipmentSlotFingerprint[]` array; each entry names the slot + expected child index + expected size range for backtracking.

### Highlight rendering

**RenderContext addition:**
```csharp
public readonly record struct EquipmentHighlightSpec(RawRectF SlotRect, uint BorderColor);
IReadOnlyList<EquipmentHighlightSpec>? EquipmentHighlights = null,
bool CharacterPanelOpen = false,
```

**RadarApp integration** (world tick, before snapshot publish):
- If character panel is open (from `Poe2Character.ReadEquipmentSlotRects` returning non-empty), for each slot:
  - Look up the `InventoryItem` from `_inventorySnapshot.EquipmentBySlot[slot]`.
  - If null (empty slot), skip.
  - Call `_itemFilterEngine.Match(item)`.
  - If any filter matches, add `EquipmentHighlightSpec(slotRect, winnerColor)`.
- Publish list on `RenderContext.EquipmentHighlights`.

**New `OverlayRenderer.DrawEquipmentHighlights`:**
- Iterate `ctx.EquipmentHighlights`, stroke a 3px colored border around each `SlotRect`.
- Call from main draw path only when `ctx.CharacterPanelOpen == true` (so we don't waste cycles when the panel is closed).

### Behavior

- Open the character panel in-game → any equipped item matching a filter gets a colored border around its slot.
- Close the panel → borders disappear (nothing to draw).
- Change gear → next world tick recomputes.

### Non-goals

- No hover tooltip augmentation (LO deferred the "replacement tooltip" ask).
- No flask row highlighting in v0.31 (add to enum but leave detection unwired until v0.32; simpler probe scope).
- No score/priority tag beside the border (border color alone communicates match).

## 13. Testing strategy

**Unit tests:**
- `ItemFilterEngineTests` — 10-15 xUnit tests covering:
  - Empty filter list → empty match.
  - Single presence-only requirement (op `>= 0`) → matches every item with the stat.
  - Threshold requirement — passes at threshold, fails below.
  - AND of two requirements — both must pass.
  - Priority ordering — higher priority filter's color wins.
  - Scope constraint — prefix-only requirement rejects suffix mods.
  - MaxTier constraint — rejects mod above tier cap.
  - Multiple ops — `>=`, `<=`, `==`, `between`.
- `ItemFilterStorageTests` — JSON load/save round-trip; missing file → empty engine; malformed file → empty engine + Console.Error message; preset copy on first-run.
- `Poe2CharacterTests` — Reads from a captured RawUiSnapshot fixture (analogous to the existing terrain fixture tests); asserts slot rects match expected values.

**Integration tests:**
- Dashboard editor: manual smoke pass — create filter, add requirements, save, reload, edit, delete, restore-presets.
- End-to-end: ground item drop with matching affixes shows colored border in `/state` inspection.

**Compliance gates (blocking):**
- `scripts/compliance-gate.ps1` must pass (no `WriteProcessMemory`, no `SendInput`, no `Marshal.Write*` in new code).
- `scripts/scrub-strings.ps1 -SelfTest` must pass (no internal-tooling references).
- `scripts/attribution-sentinel-gate.ps1` (Draft) must pass.

## 16. Fingerprint schema note (implementation clarification)

The `EquipmentSlotFingerprint[]` array in Section 10 uses this shape (locked here to avoid ambiguity during plan-writing):
```csharp
public readonly record struct EquipmentSlotFingerprint(
    EquipmentSlot Slot,          // enum value the fingerprint resolves to
    int ExpectedChildIndex,      // best-guess child index (starting point for BFS)
    float ExpectedRelX,          // relative-to-panel x normalized 0..1 (fingerprint)
    float ExpectedRelY,          // relative-to-panel y normalized 0..1 (fingerprint)
    float ExpectedRelW,          // expected width normalized 0..1
    float ExpectedRelH);         // expected height normalized 0..1
```
Baked from a live probe. Backtracking: if `ExpectedChildIndex` doesn't match the relW/relH fingerprint within ±10%, scan siblings until a match is found or the sweep exhausts. Report `null` for un-fingerprinted slots (partial detection tolerated).
