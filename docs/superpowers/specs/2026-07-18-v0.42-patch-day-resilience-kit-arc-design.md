# v0.42 "Patch-Day Resilience Kit" — Arc Design

**Status:** Draft, autonomous authorship on 2026-07-18 following 12-family / 159-offset audit ([[phase 1 workflow w5718gr2r]] + [[phase 2 workflow womowpxj2]]). Awaits LO review + approval.

**Reader thesis:** *POE2GPS survives the next patch day without a scramble.*

## Design principle

Every load-bearing hex offset in the memory-read code path becomes **observably drifted** (via a `/api/probe/*` diagnostic) or **automatically recoverable** (via a signature-based auto-heal), applied in a repeatable pattern so future patch-day work is a payload-paste + a one-line offset update instead of a multi-hop hotfix chain.

The current session cut **v0.41.3 / v0.41.4 / v0.41.5** as three sequential atlas hotfixes because the first two were speculative offset guesses without diagnostic surface — the audit shows the entire codebase carries the same class of exposure across **8 of 12 offset families**. v0.42 generalizes the pattern established by `Life.EnsureVitalOffsets` (auto-heal) and `atlasProbe` (diagnostic) into shared infrastructure that eight domain-specific beads consume.

## The 8 flagships (+ 1 scaffold)

### Bead 0. Shared Drift-Probe Scaffold (chain head)

**Reusable primitives every other bead consumes:**

- `ProbeSample<T>` record — canonical shape for offset-sweep results (offsetHex, target value or read-fail, plausibility bool)
- `HealedOffsetCache<TSymbol>` — keyed by `Poe2Offsets` symbol name; per-session persistence to `config/healed-offsets.json`; **loud console log on relocation** so operators see drift when it happens
- `/api/probe/*` endpoint helper in `ApiServer.cs` — JSON-serializes any `ProbeSample` list via a common route base with consistent loopback-Host gating
- `OffsetHealthMonitor` extension — canonical shape for "silent-critical" verdicts (`X > 0 && observed == 0 for > Ns`), mirroring the existing terrain-empty pattern at `OffsetHealthMonitor.cs:90-94`

Size: `medium` (~200 LOC). Chain head — all 7 subsequent beads depend on it.

## Locked architectural decisions (READ BEFORE ANY BEAD DISPATCH)

**1. `ProbeSample<T>` API** (locked, defined in Bead 0):
```csharp
namespace POE2Radar.Core.Diagnostics;

public readonly record struct ProbeSample<T>(
    string OffsetHex,       // e.g. "0x1B0"
    string TargetAddr,      // hex string of read address
    T? Value,               // decoded value, or default if read-fail
    string? ReadFailReason,  // null on success, e.g. "target not user-mode"
    bool PassesSignature);  // decoded value fits the domain-specific plausibility gate
```

**2. `HealedOffsetCache<TSymbol>` API**:
```csharp
public static class HealedOffsetCache
{
    public static int Resolve(string symbolName, int configuredOffset);
    public static void SetHealed(string symbolName, int healedOffset);  // logs loud + persists
    public static bool WasHealed(string symbolName);
}
```
Persistence: `config/healed-offsets.json` — schema `{schemaVersion: 1, healed: [{symbol, configured, healed, healedUtc}]}`.

**3. Endpoint naming convention (locked)**:
- `/api/probe/<family>` — new v0.42 endpoints
- Existing `/api/entity-probe` + `/api/atlas` retain their names for continuity
- All loopback-Host-gated (raw pointers in response)

**4. Loud-on-heal logging**: on any auto-heal, log `[OFFSET-HEAL] <symbol>: configured=0xXX → healed=0xYY` to stdout AND write to a rolling `healed-offsets.log` file. Users pinging LO on Discord about weird behavior can grep this to diagnose.

**5. HealthState surface**: every family exposes at least one "silent-critical" verdict via `OffsetHealthMonitor` so the dashboard's Diagnostics tab surfaces drift as an amber/red badge before the user has to notice via missing feature.

**6. NO worker git/bd operations. PRESERVE ALL OTHER BEHAVIOR. xUnit only.**

## Known risks + mitigations

1. **Bead 2 (Entity chain root) is complex + touches the render hot path.** LOCKED-DECISIONS block explicitly names the two struct bases (EntityDetails vs. reached-via-ComponentLookUpPtr) — the workflow decomposer-drift memory (`feedback_workflow_decomposer_drift`) says this is essential.

2. **Bead 5 (Flags toggle-diff) requires human-in-loop (user opens/closes atlas between two probe calls).** Later bead in v0.42.x could add auto-heal from the identified offset; v0.42 ships diagnostic-only per the audit.

3. **`healed-offsets.json` persistence risk.** If a healed offset ships to a subsequent game patch that shifts BACK to the configured value, the healed value becomes wrong. Mitigation: heal cache entries include `healedUtc`; on `Poe2Live` startup, invalidate any entry older than 30 days OR whose configured offset still passes signature gate on cold-load.

4. **Diagnostic endpoint proliferation.** 9 new endpoints. Mitigation: all under `/api/probe/*` prefix so future consolidation into a single `/api/probe` unified endpoint is easy.

5. **Ripple risk on `OffsetHealthMonitor`.** Every bead adds a hook. If one bead's hook is over-aggressive, other beads' hooks might be masked. Mitigation: each bead's hook has an independent HealthState channel (not a shared verdict).

## What v0.42 ships (release-notes shape)

**"Patch-Day Resilience Kit — the overlay survives PoE2 patches without a scramble."**

- 🩺 **Every critical memory read now has a diagnostic.** Nine new `/api/probe/*` endpoints report exactly which internal game-memory offsets each subsystem uses, whether they're still valid, and what alternates look plausible if not. When the game patches and something breaks, one `curl` + one screenshot tells LO which offset moved.
- 🔧 **Six subsystems auto-heal on drift.** AreaInstance high fields, Entity chain root, AtlasGraph routing, RuneStation monolith chain, Item rarity, and Render position all now signature-scan for alternate offsets when the configured one reads garbage. Healed offsets cache per-session and log loudly.
- 🚨 **Health-state red badges.** The dashboard's Diagnostics tab surfaces "silent-critical" states (e.g., 30s in a zone with 30 monsters visible but 0 rarities > Normal detected) as red badges before the user has to notice.
- 📝 **Loud logging on any heal event.** `[OFFSET-HEAL]` lines in the console + `healed-offsets.log` file. Grep-friendly for support.
