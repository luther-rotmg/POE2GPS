# POE2GPS v0.2.0 — Fast-Follows (Console Glow-Up · Community Contribute · Tier-Aware God-Roll)

**Date:** 2026-06-23
**Status:** Design — approved direction, pending spec review
**Targets release:** v0.2.0

## Why

Three fast-follows from the v0.1.9 cycle, plus the user's original "beautify the cmd window" ask. Built
as one bundle, sequenced **console → contribute → tier-god-roll**. The **in-game gear-score overlay**
stays out of scope — it needs a live RE session to locate the inventory-panel UI element.

## Global constraints (inherited by every task)

- **.NET 10, `net10.0-windows`, x64.** `TreatWarningsAsErrors=true`, `Nullable=enable` — 0 warnings, 0 errors.
- **Strictly read-only / GGG-compliant.** No `SendInput`/`PostMessage`/`keybd_event`/`mouse_event`, no
  `WriteProcessMemory`/`VirtualProtectEx`/`CreateRemoteThread`/injection, no pricing. `scripts/compliance-gate.ps1`
  must print PASS. (The console VT-enable uses `GetStdHandle`/`GetConsoleMode`/`SetConsoleMode` — our OWN
  console, not the game — which the gate does not scan for; confirm PASS.)
- **Preserve anti-detection.** The console **window title stays neutral/randomized** (`myName`); the
  "POE2GPS" banner is **body text only** (not an enumerable window artifact). No new identifying data on
  any wire.
- **Syncability.** New logic in new/owned files with thin hooks into the high-churn shared files
  (`Program.cs`, `RadarApp.cs`, `ApiServer.cs`, `DashboardHtml.cs`, `RadarSettings.cs`, `GearScorer.cs`).
- Tests reference `POE2Radar.Core` only; pure logic unit-tested there.
- **Data provenance:** RePoE PoE2 export (`repoe-fork.github.io/poe2`, MIT; GGG game data) — vendored
  frozen snapshot, credited, like the existing embedded tables. No PoB/GPL Lua.

## Doc fix (folded in)

CLAUDE.md and the v0.1.9 spec reference `resources/poe2-data/regenerate.py` as an existing generator —
**it was never committed** (only `merge_atlas_packs.py` exists). Part 3 authors the range generator
fresh (as a C# Research probe); a one-line CLAUDE.md correction notes the embedded RePoE tables are
generated by the `POE2Radar.Research --gen-*` probes, not a mythical `regenerate.py`.

## Part 1 — Console glow-up

### What it does

Replaces the plain `myName` / `===` banner + the two plain hotkey lines with a themed console: a
"POE2GPS" ASCII-art banner (body only), a deterministic version line, color-themed startup sections, and
a **static themed hotkey block** listing the real current hotkeys. No overlay/render changes.

### Design

New `src/POE2Radar.Overlay/Overlay/ConsoleTheme.cs` — a small static console helper:
- **VT enable:** on first use, `GetStdHandle(STD_OUTPUT_HANDLE)` → `GetConsoleMode` →
  `SetConsoleMode(... | ENABLE_VIRTUAL_TERMINAL_PROCESSING)`. If it fails (redirected/legacy), set a
  `_noColor` flag and emit plain text (graceful fallback). ~12 lines of P/Invoke.
- **Palette:** a small set of VT color constants (a cohesive gold/cyan/dim-grey theme matching the
  dashboard's gold accent). Helpers: `Banner()`, `Section(label)`, `Kv(key, value)`, `Hotkeys()`,
  `Ok/Warn/Accent(text)`. All no-op-to-plain when `_noColor`.
- **`Banner()`** prints a multi-line "POE2GPS" ASCII-art header (a compact figlet-style block, embedded
  as a `const string[]`), a tagline ("read-only PoE2 GPS overlay"), and the version line
  `v{UpdateChecker.Current}` read **synchronously** from the assembly (deterministic, up front).
- **`Hotkeys()`** prints a boxed, themed block of the real hotkeys: `F12` dashboard · `F6/F7` route/clear
  · `F9` quit · `F10` (Atlas) inspect/route · `Ctrl+Alt+]`/`[` cycle target · `Ctrl+Alt+1–9`/`0`
  slot/clear · `Ctrl+Alt+M` / `L3+R3` toggle nav menu. (No F8.)

**Hooks:**
- `Program.cs` (banner ~30-33): `Console.Title = myName;` (UNCHANGED — neutral) then `ConsoleTheme.Banner();`
  (replaces the `myName`/`===` lines). Theme the "Attached to … (PID …)" and "Running…" lines via
  `ConsoleTheme.Section`/`Ok`.
- `RadarApp.cs` ctor (256-537): wrap the Settings / Entity-names / Hidden / API summary lines as themed
  `Section`/`Kv` lines; replace the two plain hotkey lines (~520-523) with `ConsoleTheme.Hotkeys()`. The
  async update-check line (~524-537) stays but uses `ConsoleTheme.Accent/Warn` (update-available
  highlighted). Keep `Bootstrap.cs` scan lines as dim `Section` text.

### Anti-detection

`Console.Title` is left as the randomized `myName`. The ASCII art is stdout body only. No new window
class/title/process artifact. (Memorialized: [[ryan-poe2gps-preferences]] — keep full anti-detection.)

### Compliance

Console-only; `SetConsoleMode`/`GetStdHandle` are not input-emission/process-write APIs (they configure
our own console). Gate stays green.

### Tests

Console output is not unit-tested directly. The one piece of pure logic — the plain-text fallback +
the hotkey-list content — can be exercised by a `ConsoleTheme.RenderHotkeysPlain()`-style pure method if
trivially extractable; otherwise covered by the manual live check (banner + themed hotkeys render; colors
degrade gracefully if VT unavailable). Keep any pure helper in a form a Core test could call only if it's
clean to do so; do not contort the design for a test.

## Part 2 — Community Contribute (one-click → your Worker → your repo)

### What it does

A one-click "Contribute" that uploads the user's **non-identifying** Entity-Atlas pack
(`{names, objectives}`) to a Cloudflare Worker the user deploys, which files it to
`luther-rotmg/POE2GPS` — with a green "thank you" toast. Ships **inert** until the user configures the
Worker URL (then the button activates; otherwise the existing GitHub-issue-template link remains).

### Design — in-client

- **Setting:** `RadarSettings.ContributeUrl` (string, default `""`). Round-trips via a new flat string
  case in `/api/settings` (add a `TryString` helper beside `TryBool/TryInt/TryFloat`; GET projection adds
  `contributeUrl`; a Settings text input `data-set="contributeUrl"`). When empty → contribute-upload off.
- **New route `POST /api/contribute`** (loopback-gated, like `/api/entity-atlas/import`): server-side it
  reads the export pack (`{names = _entityNames.All, objectives = _objectives.All}` — the same payload as
  `/api/entity-atlas/export`), and if `ContributeUrl` is set, POSTs it to that URL via `HttpClient`
  (outbound, like `UpdateChecker`), returning `{ok, status}` or an error. **No CORS** (the browser hits
  same-origin localhost; the outbound POST is server-side, and the Worker URL never reaches the browser).
  Caps the body and never includes identifying data (the pack is verified non-identifying at source).
- **Dashboard (Entity Atlas tab):** the `eaContribute` button becomes: if a Worker URL is configured
  (from `/api/settings`), a **one-time confirm** ("shares your discovered names/POIs publicly — no
  character data") then `POST /api/contribute` → green toast `✓ contributed — thank you!` (reuse the
  `.saved/.show` idiom; add `savedMsgEa` + `flashEa()`). If no URL configured, keep the current
  `window.open(issue-template)` fallback. Tooltip explains the two modes.

### Design — the Worker (user deploys once; I write it + the guide)

New `cloudflare-worker/` dir (NOT shipped in the overlay; outside the compliance gate):
- `worker.js` — handles `OPTIONS` (CORS preflight, in case of future browser-direct use) + `POST`;
  validates the body is `{names: object, objectives: array}`, **rejects** any payload containing
  identifying keys (defense-in-depth), caps size (e.g. 256 KB), rate-limits per IP (Cloudflare), and
  creates a GitHub issue in `luther-rotmg/POE2GPS` (title + the pack in the body, labeled
  `community-pack`) via the GitHub API using a **fine-grained token held as a Worker secret**
  (`GITHUB_TOKEN`, scoped to issues:write on that repo only). Returns `200`/`400`/`429`.
- `wrangler.toml` + `cloudflare-worker/README.md` — deploy steps: `wrangler deploy`,
  `wrangler secret put GITHUB_TOKEN` (the user creates a fine-grained PAT scoped to issues on
  POE2GPS), then paste the Worker URL into the dashboard Settings.

### Privacy / safety

Upload is **user-initiated** (button click) + first-click confirm; payload is **non-identifying**
(verified: `names` = metadata→label map, `objectives` = matcher rules — no character/account/address).
The Worker URL is user-configured; nothing is sent by default. The token lives only in the Worker secret.

### Compliance

In-client: a settings string + a loopback-gated route that does an outbound POST (like UpdateChecker) of
a non-identifying pack — no game input/write. The Worker is infra, not shipped source.

### Tests

The in-client path is config + an outbound-forward route; covered by the manual check (configure URL →
button activates → POST round-trips → toast). If the payload-shape guard is extracted as a pure check, a
small Core test can assert it rejects an identifying key and accepts `{names, objectives}`.

## Part 3 — Tier-aware god-roll (%-of-max-roll + T#/N)

### What it does

Shows, per affix, **how good the roll is**: `% of max` (the rolled value vs the mod's max) and the mod's
**tier** (`T2 / 8`). A god roll becomes "near the top of the real GGG range / high tier", not just high
weight×value. Display + data only; the existing weight-based score/threshold is unchanged for v0.2.0.

### Design — data (offline, from RePoE; license-clean)

- **Vendor:** `resources/poe2-data/repoe-mods.min.json` — a frozen snapshot of
  `repoe-fork.github.io/poe2/mods.min.json` (MIT/GGG-data; credited). Each mod object carries
  `stats[].{id,min,max}`, `groups[]`, `generation_type`, `domain`, `required_level`; keyed by the **same
  mod id** as our `poe2_mod_stats.json` (1:1 join).
- **New Research probe `--gen-ranges`** (dispatched ABOVE `AttachToPoE`, like `--gen-weights`): reads the
  vendored snapshot and emits embedded `src/POE2Radar.Core/Game/poe2_mod_ranges.json`. For each mod id:
  `{ stats: [{id, min, max}], tier: N, tierCount: M }` where **tier is DERIVED** — bucket mods on
  `(groups[0], generation_type, domain)`, rank by `required_level` **descending** so **T1 = best**
  (highest-level/strongest), `tierCount` = bucket size. Per-stat ranges are stored aligned to the mod's
  stat ids (so multi-stat/hybrid mods keep separate ranges). Minified output.
- **New Core `ModRanges` loader** (`src/POE2Radar.Core/Game/ModRanges.cs`) — loads the embedded
  `poe2_mod_ranges.json` once (mirroring `StarterWeights`/`ItemModTranslator`). Exposes
  `bool TryGet(string modId, out ModRangeInfo info)` where `ModRangeInfo` carries the per-stat
  `{min,max}` list + `Tier`/`TierCount`. Returns false for unknown mods (graceful degradation).

### Design — read/scorer integration

The gear read already has each affix's `modId` + rolled `Values` (`RawAffix`), but the scorer's `Affix`
drops the modId. To compute %-of-max + tier we thread the modId through:
- `Core/Gear/GearScorer.cs`: add `ModId` to `Affix` (`Affix(StatLine, StatIds, Value, ModId)`) and to
  `AffixContribution` (carry it through `Score`, scorer logic otherwise unchanged — it still scores by
  weight/norm). The `WorldTick` affix build passes `ra.ModId`.
- `RadarApp.GearJson`: for each affix, look up `ModRanges.TryGet(modId)` → compute `pctOfMax = round(
  value / max * 100)` against the matching stat's max (guard `max<=min` / `min==max` → fixed roll, report
  `100` / no-range), and surface `tier`/`tierCount`. Add `pctOfMax`, `tier`, `tierCount` to the affix
  JSON. Absent mod → omit (value-only).
- **Item-level "best tier" / god-roll hint:** GearJson can add an item-level `bestTier` (min tier across
  the item's affixes) and keep the existing `godRoll` (weight threshold). A small UI cue (e.g. "T1" badge)
  marks a top-tier roll. Scope guard: do NOT change the score formula in v0.2.0 — tier/pct is
  informational; a tier-weighted score is a later option.

### Design — dashboard

In the Gear tab affix rows (and optionally the grid cell title), show `· 87% · T2/8` next to each affix
(when range data exists). Color the %-of-max (e.g. ≥90% gold). Keep it compact.

### Compliance

All additions are read-only (embedded data + a pure lookup; the read already happens). No input/write.
The probe is dev-only (Research). Gate green.

### Tests

- `--gen-ranges` tier derivation: a `Core`-side pure helper `TierDeriver` (bucket + rank) unit-tested —
  e.g. an `IncreasedLife` family ranks so the highest-required-level mod is `T1` and `tierCount` = family
  size.
- `ModRanges` loader: the embedded `poe2_mod_ranges.json` parses, is nonempty, and a known mod (e.g.
  `IncreasedLife9`) returns a `{min,max}` with `Tier`/`TierCount` > 0.
- `GearScorer`: `Affix`/`AffixContribution` carry `ModId` (back-compat: existing tests updated to the new
  arity); a %-of-max guard test (min==max → no div-by-zero).

## Out of scope (separate tracks)

- **In-game gear-score overlay** — needs a live RE session (find the inventory-panel UI element via the
  Research `--devtree`); the `poe2-craft-inspector` clipboard route is a possible alt input. Its own spec
  after discovery.
- **Tier-WEIGHTED scoring** (folding %-of-max/tier into the numeric score) — v0.2.0 ships tier/pct as
  display only; weighting it is a later option once the data proves out live.

## Success criteria

1. The console shows a themed "POE2GPS" banner (body only; window title stays neutral) + a themed static
   hotkey block with the real hotkeys; colors degrade gracefully without VT.
2. With a Worker URL configured, the Contribute button uploads the non-identifying pack and shows the
   green "thank you" toast; with none configured, the issue-template fallback remains. The Worker +
   deploy guide exist.
3. Gear affixes show `% of max` + `T#/N` when range data exists (degrade to value-only otherwise),
   sourced from RePoE (no PoB Lua). The weight-based score is unchanged.
4. Build 0W/0E, all tests pass, compliance gate PASS, no identifying data on any wire, anti-detection
   preserved.
