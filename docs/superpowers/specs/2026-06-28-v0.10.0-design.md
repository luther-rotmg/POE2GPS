# v0.10.0 — Onboarding · Map-mechanic intelligence · Settings search · Custom keybinds — Design

**Date:** 2026-06-28
**Branch:** `feat/v0.10.0`
**Status:** Approved direction — spec for implementation

## Goal

A four-feature QoL + intelligence batch, all strictly read-only / GGG-compliant, all solo-buildable (no memory RE):
1. **Onboarding & quick-start** — a first-run welcome card + a one-click "recommended setup".
2. **Map-mechanic radar intelligence** — count nearby league mechanics in the Zone summary + an optional "mechanic nearby" audio cue.
3. **Settings search & tidy-up** — a search/filter box + collapsible cards in the Settings tab.
4. **Custom keybinds** — remappable keyboard hotkeys from the dashboard.

## Compliance invariant

Everything here is read-only / output / config. Custom keybinds only changes *which* keys we poll via `GetAsyncKeyState` — we still never send input. The recommended-setup + keybind + first-run writes are loopback-gated config. Compliance gate + scrub stay green; README badge stays `0.5.4`.

## Feature 1 — Onboarding & quick-start

### 1.1 First-run flag + card
- `RadarSettings.FirstRunSeen` (bool, default **false**). On first launch (fresh config), it's false.
- The dashboard shows a prominent **"Quick start"** card at the top of the default view while `!firstRunSeen`. It contains:
  - The 3 essentials: run `Overlay.exe` as Administrator · be in a zone · **F12** toggles this dashboard.
  - A compact **hotkey + feature cheat-sheet** (the current binds + what the headline features do).
  - An **"Apply recommended setup"** button and a **"Dismiss"** button.
- **Dismiss** → `POST /api/quickstart/dismiss` (loopback-gated) sets `FirstRunSeen=true` + saves; the card hides. (A small "Quick start" link remains available to re-open it.)

### 1.2 Recommended setup
- A curated, **non-surprising** set of safe QoL toggles applied server-side via `POST /api/quickstart/apply` (loopback-gated): enable the **Zone summary** panel, ground-item labels for **Uniques + Currency**, and **HP bars for Rare + Unique**. (No audio — it stays off by default by design; nothing that changes the radar's core look.) The exact set lives in one server-side method so it's canonical and documented.
- Applying does **not** require the card — it's also reachable later (the re-open link), so existing users can opt in.
- We do **not** change any existing defaults (upgraders keep their config untouched; the flag just controls the card).

## Feature 2 — Map-mechanic radar intelligence

### 2.1 Mechanic pattern table (Core, testable)
`POE2Radar.Core/Game/MechanicPatterns.cs` — a static table mapping a mechanic name → metadata substrings, covering: **Strongbox, Shrine, Breach, Expedition, Ritual, Delirium** (the metadata families already matched by the display-rule mechanics). API: `MechanicPatterns.Classify(metadata) → string?` (the mechanic name or null) and `MechanicPatterns.Names`. Pure string matching → **unit-tested** (known metadata paths classify correctly; unrelated paths → null; case-insensitive contains).

### 2.2 Zone summary mechanic counts
Extend the v0.9.1 Zone summary aggregation (world thread, over `_entities`) to also tally per-mechanic counts via `MechanicPatterns.Classify(e.Metadata)`. The panel renders a mechanic row only when its count > 0 (keeps it compact in zones with no mechanics). Carried through the existing `ZoneSummaryBundle` / `RenderContext.ZoneSummary` (extend the struct with a small mechanic-count list or fixed fields).

### 2.3 "Mechanic nearby" audio cue
A new edge-triggered audio event `AudioAlertMechanic` (bool, default true under the master gate, which stays off) — fires when an entity classified as a mechanic enters the alert radius, reusing the v0.9.x cue infra (a 4th cue + tone setting `AudioToneMechanic`, default a distinct tone; dedup `HashSet<uint>` cleared on area change; the radius + cooldown mirror the monster cue). Dashboard: a toggle + tone select in the Audio card; a count-row toggle isn't needed (mechanic rows auto-show when present).

## Feature 3 — Settings search & tidy-up

### 3.1 Search box
A text input pinned at the top of the Settings tab. On input, filter the Settings cards: a card stays visible if its heading or any of its row labels contains the query (case-insensitive); others are hidden. Clearing the box restores all. Pure dashboard JS — no backend, no settings.

### 3.2 Collapsible cards
Each Settings card heading becomes a click target that toggles a `collapsed` class (hides the card body, keeps the heading). The collapsed set persists in `localStorage` (per card, by a stable card id). A small chevron indicates state. The Status / Quick-start cards stay always-expanded (they're informational). Pure JS.

(Grouping into sections is out of scope for v1 — search + collapse covers the navigability need.)

## Feature 4 — Custom keybinds

### 4.1 Keybind model (Core, testable)
`POE2Radar.Core/Input/Keybind.cs` — a `readonly record struct Keybind(bool Ctrl, bool Alt, bool Shift, int Vk)` + `Keybind.Format()` → a display string (e.g. `"Ctrl+Alt+R"`, `"F6"`) and `Keybind.TryParse(string) → Keybind?`. A `KeybindDefaults` table maps each **action** to its current default bind. Pure → **unit-tested** (round-trip parse/format; modifier combos; F-keys; unknown → null).

**Remappable actions** (the discrete single-key hotkeys): `OpenDashboard` (F12), `Quit` (F9), `AtlasInspect` (F10), `RouteNearest` (F6), `ClearRoutes` (F7), `CycleNext` (Ctrl+Alt+]), `CyclePrev` (Ctrl+Alt+[), `ToggleNavMenu` (Ctrl+Alt+M), `ResetSession` (Ctrl+Alt+R). **Fixed (not remappable):** the slot-jump digits (Ctrl+Alt+1–9/0) and the controller R3/L3 (special — they ride PoE2's stick-clicks).

### 4.2 Settings + input layer
- `RadarSettings.Keybinds` (a dictionary or a small typed object action→Keybind; defaults = `KeybindDefaults`). Round-trips via the existing JSON serializer; missing keys fall back to the default for that action.
- `RadarApp.HandleHotkeys` reads each action's key from `_settings.Keybinds` (via a resolver `Keybind For(action)`), preserving the existing special handling (the cycle hold-repeat, the menu/reset debounce). Only the *trigger key* per action becomes configurable; the behavior is unchanged.

### 4.3 Dashboard keybind editor
A **"Keybinds"** card in Settings listing each remappable action with its current bind + a **"Rebind"** button. Clicking Rebind enters capture mode: the next `keydown` is captured in JS, mapped to `{ctrl,alt,shift,vk}` (a JS-key → Windows-VK map for F1–F12, A–Z, 0–9, brackets, etc.), and `POST`ed (loopback-gated) to update that action's bind. A **"Reset to defaults"** button restores `KeybindDefaults`. Conflicts (two actions on the same combo) are flagged in the UI and rejected server-side (the later bind wins only if the user confirms; default: reject with a message).

### 4.4 Compliance
Still strictly read-only: the overlay only *reads* `GetAsyncKeyState` for the configured VKs (foreground-gated, in-game). Remapping changes which keys are read — never anything sent. No new input APIs.

## Architecture / boundaries
- New Core (pure, tested): `Game/MechanicPatterns.cs`, `Input/Keybind.cs` (+ `KeybindDefaults`).
- New/changed Overlay: `RadarSettings` (`FirstRunSeen`, `Keybinds`, `AudioAlertMechanic`/`AudioToneMechanic`); `ApiServer` (`/api/quickstart/{dismiss,apply}`, keybind GET/POST, mechanic-audio + zone-mechanic settings); `RadarApp` (mechanic counts in the zone aggregation, mechanic audio cue, `HandleHotkeys` reads `Keybinds`); `OverlayRenderer` (mechanic rows in `DrawZoneSummary`); `RenderContext`/`ZoneSummary` (mechanic counts); `DashboardHtml` (Quick-start card, Settings search + collapse, Keybinds editor, mechanic audio toggle).
- No Core→Vortice; `Program.cs` console title untouched; all writes loopback-gated.

## Testing
- **Core unit tests:** `MechanicPatterns.Classify` (known/unknown/case), `Keybind` parse/format round-trip + defaults. Both pure → fit the Core-only test project.
- **Build + review (Overlay):** the dashboard cards/search/editor, the quickstart endpoints, the mechanic counts/audio, and the `HandleHotkeys` rewire — no Overlay harness; behavior-preserving where applicable, verified by build 0W/0E + full suite + compliance/scrub gates + per-task & final reviews. In-game smoke (rebind a key, mechanic counts/cue, recommended-setup) deferred to the validation session.

## Out of scope (YAGNI)
- Remapping the slot-jump digits or controller buttons.
- A community preset/keybind gallery; cloud sync.
- Settings section-grouping (search + collapse is enough).
- Per-mechanic count toggles (mechanic rows auto-show when present).

## Version
Ships as **v0.10.0** (feature-sized, matches the project's x.Y.0-for-features convention). README badge stays `0.5.4`.
