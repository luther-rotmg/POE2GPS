# v0.41 "Supporter Bundle" — Arc Design

**Status:** Draft, awaiting LO review. Design approved verbally 2026-07-17.

**Reader thesis:** *Focused play. Your loadout follows your zone.*

## Design principle

Every feature in this arc is a **new capability that never existed before supporter-tier**. The free-tier experience remains byte-identical to today — nothing is removed, degraded, or newly-gated. Supporter unlock adds NEW behavior; it doesn't take anything away.

This distinguishes the arc from a "freemium" pattern where features are yanked behind a paywall. Prior supporter surface (v0.35 dashboard palettes + v0.38 Color Forge / Preset Gallery / Import) followed the same principle and is treated as reference.

## The 4 flagships

### A. Focused Radar Filter

Per-zone entity **whitelist + blacklist** with two runtime effects:

- **Blacklist runs at the renderer entity-loop early-continue**: matching entities are never Y-sorted, never allocated in per-tick draw structures, never drawn. Real per-frame savings — a hideout with ~40 decorative NPCs / critters / vendor pets typically produces measurable frame-time reduction when blacklisted.
- **Whitelist elevates matches above the standard Y-sort**: a rule-matched entity always renders above others regardless of world Y-position. Boss / rare / valuable-drop entities become unmissable through visual clutter.

**Storage:** `config/radar-filters.json` — one preset per zone-code prefix. Wildcard `*` acts as default fallback. Shape:
```json
{
  "schemaVersion": 1,
  "presets": [
    { "match": "*_town",     "whitelist": ["Metadata/NPC/Traders/*"], "blacklist": ["Metadata/Effects/*", "Metadata/Monsters/Ambient/*"] },
    { "match": "T17_*",      "whitelist": ["Metadata/Monsters/Uniques/*"], "blacklist": [] },
    { "match": "*",          "whitelist": [], "blacklist": [] }
  ]
}
```

**Selection:** presets iterate in **array order**; **first matching preset wins** on zone entry. Wildcards use the same `*`-glob shape as v0.39 Rule Engine selectors for consistency. Cap: **20 presets** (dashboard warning at 15), **50 metadata patterns** per whitelist/blacklist.

**Dashboard editor:** new tab authoring these presets — one row per preset, with match-pattern text input + two chip lists (whitelist / blacklist), add-from-current-zone-entities helper button.

### B. Zone-Aware Overlay Layouts

3–5 saveable HUD **layout presets** that auto-swap on zone entry. Each preset captures panel visibility + positions across the overlay's `data-panel-id` panels.

**Example presets:**
- `town` — hides drop timeline, boss HP, XP chart. Vitals + minimal chip strip only.
- `map` — full loadout (default).
- `boss-arena` — vitals + boss HP + telegraph indicators only; everything else hidden.

**Storage:** `config/overlay-layouts.json`. Shape:
```json
{
  "schemaVersion": 1,
  "presets": [
    { "name": "town",       "match": "*_town",     "panels": { "dropTimeline": {"visible":false}, ... } },
    { "name": "boss-arena", "match": "Boss_*",     "panels": { ... } },
    { "name": "default",    "match": "*",          "panels": { ... } }
  ]
}
```

**Dashboard editor:** "capture current layout → save as → assign zones" workflow. Same `ZoneCodeMatcher` wildcard shape as Feature A for consistency; **first matching preset wins** in array order. Cap: **10 layouts** (dashboard warning at 8).

**Panel enumeration:** overlay renderer exposes a `data-panel-id` attribute contract for every draggable/hideable panel. This bead codifies that contract (some panels already have it; audit + backfill needed).

### C. Custom Auto-Nav Destinations

Supporter defines **named A\* endpoints per zone** at specific grid coordinates. Free tier keeps every existing built-in curated landmark; supporter adds their own named destinations on top.

**Example:**
- `T17-Necropolis / chest room` at `(145, 220)`
- `Delirium-Mirror / altar` at `(50, 180)`

**Overlay UX:** chip in the corner of the overlay reads `→ chest room` when a saved destination exists in the current zone. Click → routes via the existing landmark A\* render path (already used by curated landmarks).

**Storage:** `config/nav-destinations.json`. Shape:
```json
{
  "schemaVersion": 1,
  "destinations": [
    { "zoneCode": "T17_Necropolis", "name": "chest room", "x": 145, "y": 220 },
    { "zoneCode": "Delirium_Mirror", "name": "altar",    "x":  50, "y": 180 }
  ]
}
```

**Dashboard editor:** table of destinations, "capture current player position as destination" helper button, per-row name / zone / x / y edit fields, delete button. Cap: **50 destinations** (dashboard warning at 40).

**Zone matching (intentional divergence from A + B):** destinations use **exact `zoneCode` equality**, NOT wildcard patterns. A named "chest room" location only makes sense for a specific zone — wildcards would produce misleading routes. `ZoneCodeMatcher` is not used here; simple string equality.

**Existing landmark path:** overlay's A\* landmark render already takes `(zoneCode, name, gridPos)` — this feature just injects supporter-authored records into the same iteration alongside curated ones.

### D. Session Stat Widget

Small persistent widget on the **overlay itself** (not the dashboard) showing metric chips the supporter picks. 6 built-in metrics; user picks which chips render.

**Built-in metrics:**
- `drops` — count of notable drops this session
- `xp-gained` — XP delta since session start
- `bosses-killed` — count of boss kills this session
- `deaths` — session death count
- `time-in-zone` — mm:ss since current zone entry
- `avg-map-clear-time` — rolling average over the **last 10 map clears** (falls back to "—" if fewer than 3 completed this session)

**Storage:** `config/session-widget.json`. Shape:
```json
{
  "schemaVersion": 1,
  "position": { "x": 20, "y": 20 },
  "chips": ["drops", "xp-gained", "time-in-zone"]
}
```

**Data source:** all 6 metrics derive from existing `SessionTracker` fields + `/state` — no new game-memory reach.

**Dashboard editor:** checkbox for each of the 6 metrics + drag-position preview.

## Shared infrastructure

Write once, reused by all 4 flagships + retrofit into existing v0.35 palette gate:

### `SupporterGate` (static helper)

Canonical replacement for the ad-hoc `s.isSupporter` checks scattered in `dashboard.js` and the existing `applySupporterCosmetics` code path. Single source of truth. Testable in isolation.

Public API:
```csharp
public static class SupporterGate {
    public static bool IsSupporter { get; }  // cached from settings
    public static void RefreshFromSettings(RadarSettings settings);
}
```

JS mirror:
```js
window.__supporterGate = { isSupporter: () => bool, refresh: async () => void }
```

### `ZoneCodeMatcher`

Wildcard-prefix matching reused by A + B (and any future zone-aware feature). Same `*`-glob shape as v0.39 Rule Engine selectors.

```csharp
public static class ZoneCodeMatcher {
    public static bool Match(string pattern, string zoneCode);
    // Match("*_town", "G1_town")       => true
    // Match("T17_*",  "T17_Necropolis") => true
    // Match("*",      anything)         => true
    // Match("Boss_Arena", "G1_town")    => false
}
```

### `SupporterHint` (UI component)

One styled block that renders in-place when a non-supporter tries to interact with a supporter feature editor. NOT a modal, NOT a toast — an inline card where the editor would be.

Copy: *"Save your Ko-fi code in Settings to unlock this."* Link to the Settings > Support section.

Applied consistently across all 4 feature dashboard editors + the existing v0.35 palette selector (retrofit — currently uses ad-hoc gate).

### Common JSON envelope shape

All 4 features use `{schemaVersion: 1, ...}` envelope + whole-file atomic write via `.tmp` + `File.Move` (mirrors v0.39 `RulesFileStore`). NOT JSONL — none of these grow unbounded in size.

## Dispatch sequencing

- **Wave 1** (chain head): S1 → S2 → S3
- **Wave 2** (parallel-eligible after S1/S2/S3 land): A1, B1, C1, D1 (all pure `*Store` classes, no cross-deps)
- **Wave 3** (parallel per-feature): A2/A3, B2/B3/B4, C2/C3/C4, D2/D3 — cascade within each feature
- **Wave 4**: R1 (release)

## Global constraints

- **Read-only preserved** — all 4 read existing fields; no new game-memory reach
- **No telemetry** — all storage local `config/*.json`
- **Additive** — nothing existing removed, no free-tier behavior changed
- **Character-name privacy** — Feature D shows metrics only, no character name; if character-name ever surfaces it goes through loopback-gated `/api/session-widget?character=<name>`
- **Supporter check** — canonical `SupporterGate.IsSupporter` used everywhere; no scattered `s.isSupporter` checks after S1 ships
- **Fully backwards-compatible** — existing v0.35 palette gate refactored to use `SupporterGate` behind the scenes; user-visible behavior unchanged

## Known risks + mitigations

1. **Radar filter perf claim** must be measurable. Ship xUnit bench that walks a 100-entity synthetic snapshot with blacklist active vs. inactive; assert active is meaningfully faster.
2. **Panel `data-panel-id` audit** — some overlay panels may not currently have the attribute. B2 audits and backfills; discovered panels documented in the bead output.
3. **Custom nav-destination A\* seam** — the existing landmark render path might not accept dynamic-injection cleanly. C2 might need a small refactor of the landmark iterator; if scope creeps, split into C2a (iterator refactor) + C2b (injection).
4. **Session widget position drag** — existing overlay panels don't have a shared drag framework (v0.41 "Dashboard Studio" per arc parent doc); D3 uses a simple x/y config with no drag UI, defers drag-to-position to a future v0.41.x if requested.
5. **Supporter code validation cache** — `SupporterGate.IsSupporter` needs to refresh when the user changes their code. S1 subscribes to the existing settings-changed signal (grep for the pattern in current palette gate).

## What v0.41.0 ships (release-notes shape)

**"Focused play. Your loadout follows your zone."**

- 🎯 **Focused Radar Filter** — supporter tier only. Per-zone whitelist + blacklist. Blacklist skips the render walk entirely (real FPS win in town/hideout). Whitelist always draws on top.
- 🧩 **Zone-Aware Overlay Layouts** — supporter tier only. Save 3-5 HUD layouts and let them auto-swap when you zone. Town gets minimal, boss-arena gets vitals-only, maps get your full loadout.
- 📍 **Custom Auto-Nav Destinations** — supporter tier only. Name any grid position ("chest room at 145,220 in T17-Necropolis"). Overlay chip routes you there with one click.
- 📊 **Session Stat Widget** — supporter tier only. Small overlay widget with configurable chips: drops, XP, bosses, deaths, time-in-zone, avg map-clear time. Pick which show.
- Under the hood: `SupporterGate` centralizes the tier check (was scattered); `ZoneCodeMatcher` wildcard matcher shared across A + B.
- Free tier: byte-identical experience to v0.40.1. None of the above appears or activates without a valid Ko-fi code.
