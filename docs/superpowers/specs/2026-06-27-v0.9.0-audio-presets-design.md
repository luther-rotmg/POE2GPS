# v0.9.0 — Audio Alerts + Community Presets — Design

**Date:** 2026-06-27
**Branch:** `feat/v0.9.0-audio-presets`
**Status:** Approved direction (brainstorm) — spec for implementation
**Grounding:** workflow `ww99zzfvk` (audio events/loop, preset model, preset I/O pattern)

## Goal

Two user-facing features, both strictly read-only / GGG-compliant:
- **A. Audio Alerts** — short, distinct procedurally-generated tones for a few high-signal events. **Default OFF**, pinned **prominently at the top of the Settings tab**, per-event toggles.
- **B. Community Presets** — share your radar's *look* (display rules + visual styles) via **both** a copy-paste share-code **and** a `.poe2preset` file (export + import).

These are two independent subsystems sharing only `RadarSettings` / `DashboardHtml` / `ApiServer`. Build order: **Audio first** (smaller, Core-testable), then Presets.

## Compliance invariant

Audio is **output only** — no `SendInput`, no memory writes; it reads the same per-tick entity data the radar already reads. Presets only swap **visual config** (never operational/anti-detection/process settings) and are import-sanitized. The compliance gate + scrub stay green. README badge stays `0.5.4` (no offsets touched).

## A. Audio Alerts

### A.1 Tone generation + playback
- **`POE2Radar.Core/Audio/PureToneWav.cs`** — `public static byte[] Generate(double freqHz, int durationMs, double volume = 0.5, int sampleRate = 44100)`: returns a complete 16-bit mono PCM WAV (44-byte RIFF/WAVE header + samples), a sine wave with a short linear attack/decay envelope (≈8 ms each) to avoid clicks. **Pure BCL math, no Windows API → unit-tested in the Core-only test project** (assert: bytes start `RIFF`…`WAVE`, `fmt `/`data` chunks present, sample count == `sampleRate*durationMs/1000`, peak amplitude ≤ `volume*32767`, silence at `freq=0`).
- **`POE2Radar.Overlay/Audio/AudioCue.cs`** — wraps one `System.Media.SoundPlayer` over a pre-generated WAV `MemoryStream`. `Play()` is fire-and-forget (`SoundPlayer.Play()`, non-blocking; safe to call from the world thread). One `AudioCue` per distinct tone so cues can overlap (SoundPlayer can't overlap a *single* player). Constructed in `RadarApp`'s ctor; `Play()` swallows exceptions (audio must never crash the overlay).
- Three cues, distinct pitches (recommended): **monster** ~660 Hz/120 ms, **unique drop** ~880 Hz/150 ms (two quick beeps), **objective reached** ~520 Hz/180 ms.

### A.2 Events + detection (all in `WorldTick`, ~30 Hz, in-zone-gated)
A single private `CheckAudioEvents(NumVec2 player)` called once per `WorldTick` after `MaintainRoutes` (RadarApp.cs ~:1347), iterating `_entities` once. `WorldTick` only runs at `ResolveStage.Full`, so this is inherently **in-zone-gated** (no chimes at menu/loading/zoning). **No window-focus gate** — a unique dropping while you're alt-tabbed is exactly when the cue is most useful (audio output when unfocused is harmless/desirable).
1. **Rare/Unique monster nearby** — `e.Category==Monster && e.Rarity>=Rare && e.IsAlive && !e.IsFriendly && NumVec2.Distance(e.Grid,player) < AudioAlertRadiusCells`. Edge-triggered via a `HashSet<uint> _alertedMonsters` keyed on `e.Id`: fire+add on first in-range; remove when it leaves range (re-entry re-fires). Per-cue cooldown ~3 s (a `Stopwatch.GetTimestamp` gate) so entering a dense pack doesn't machine-gun.
2. **Unique item on the ground** — `e.ItemArt is not null && e.Rarity==Unique` (data already on `EntityDot` from the entity walk — **no `ScanLootLabels` needed**). Edge-triggered via `HashSet<uint> _alertedItems` on `e.Id`. Discrete drops → dedup only, no cooldown.
3. **Objective reached** — for each selected nav id (the `SnapshotSelection()` list, which already includes the Campaign-GPS / Director target since they funnel through `_selectedIds`), `TryResolveTargetGrid(id, out goal)` and `NumVec2.Distance(goal,player) < AudioArrivedCells` (≈8). Edge-triggered via `HashSet<string> _alertedTargets` on the id.
All three dedup sets are **world-thread-owned** (no locks) and **cleared in `OnAreaChanged`** (RadarApp.cs ~:1776) so each fires once per zone visit.

### A.3 Settings (flat keys — auto-bind to the dashboard with zero extra JS)
Added to `RadarSettings.cs` as plain bool/int auto-properties (round-trip automatically via the camelCase serializer):
- `EnableAudioAlerts` (bool, **default false** — the master gate; nothing plays out of the box)
- `AudioAlertRareUnique` (bool, default true), `AudioAlertUniqueDrop` (bool, default true), `AudioAlertObjective` (bool, default true) — each gated under the master
- `AudioAlertRadiusCells` (int, default 60)
Each plays only when `EnableAudioAlerts && <sub-toggle>`. Wired in `ApiServer` `/api/settings` GET (add fields) + POST (one `case` per key, mirroring the existing `hideJunk` cases).

### A.4 Dashboard card (top of Settings)
A new `<div class="card"><h3>Audio alerts</h3>…</div>` inserted in `DashboardHtml.cs` **between the Status card and the "Target cycling" card** (~:549) — i.e. the first interactive card in Settings. Each toggle is `<input type="checkbox" data-set="enableAudioAlerts">` etc. (the `[data-set]` auto-binding in `wireSettings()`/`loadSettings()` handles save+load with no per-toggle JS). A numeric input for `audioAlertRadiusCells`. A small **"Test"** button per cue → `POST /api/audio-test` `{cue:"monster|item|objective"}` (loopback-gated) so users can audition the tone.

### A.5 Audio threading/risks
`SoundPlayer.Play()` is non-blocking and called only from the single world thread → no contention; construct cues on the main thread in the ctor; never `Dispose` while playing. Dedup sets are world-thread-exclusive. If multiple events fire the same tick, all three cues are separate players so they may overlap — acceptable. `TreatWarningsAsErrors`/`Nullable`: cue fields initialized in the ctor (not `= null!`).

## B. Community Presets

### B.1 The preset model — a SEPARATE class (critical)
**`POE2Radar.Overlay/Config/PresetBundle.cs`** — a dedicated sealed class, **never** a serialized `RadarSettings` (serializing the whole settings object would carry the one-time migration guards `AbyssRuleSeeded`/`IconDefaultsApplied*`/`RuleCleanupV1`/… which, set back to false on import, would re-run migrations and corrupt the user's rules). Fields = **visual payload only**:
- `RadarStyles Styles`, `HpBarSettings HpBars`, `TerrainSettings Terrain`
- `bool ShowMonsters/ShowTerrain/ShowPlayerBlip`, `bool HpBarNormal/Magic/Rare/Unique`
- `GroundItemSettings GroundItems` (Enabled + Categories subset)
- `List<DisplayRule> DisplayRules` (the full ordered ruleset — the heart of a shared "look")
- Metadata (never applied to settings): `string Name`, `string Author`, `string AppVersion`, `int Schema` (=1), `string CreatedAtUtc` (string, not `DateTime` — avoid clock dependency)

**Explicitly EXCLUDED** (operational / identity / calibration — must never be in a preset): `ApiPort`, `ExcludeFromCapture`, `CheckForUpdates`, `EnableTargetHotkeys`, `EnableControllerCycle`, cycle timings, `FpsCap`, `NavMenuCorner`, `ContributeUrl`, `AlwaysShowOverlay`, `ScaleMul/OffX/OffY`, `AutoNavPatterns`, all `Atlas*` flags, `HighlightDynastyMaps`, `EnableDirector/CampaignGps/QuestMemory`, `EnableGearScorer`, `SessionHud`, the audio settings, and **all migration guards**.

### B.2 Codec (share-code)
**`POE2Radar.Core/Config/PresetCodec.cs`** (or Overlay — but Core makes the round-trip **unit-testable**): `string Encode(string json)` = `"POE2GPS-" + base64url(gzip(utf8(json)))`; `bool TryDecode(string code, out string json)` reverses, tolerating the prefix being absent. Unit test: `TryDecode(Encode(x)) == x`; rejects garbage; bounded output. JSON serialized with the same camelCase + `WhenWritingNull` options as `DisplayRules.Json`.

### B.3 Endpoints (server-side — keep serialization canonical)
Serialization stays **server-side** (the codebase invariant — JS never builds config). In `ApiServer.cs`, mirroring the entity-atlas pair:
- **`GET /api/preset/export`** (read-only, no gate) → `{ name, json, code }` where `json` is the live `PresetBundle` (built from `_settings` visual fields + `_displayRules.All`) and `code` is `PresetCodec.Encode(json)`.
- **`POST /api/preset/import`** (POST-only 405-guard + `IsLoopbackHost` 403-guard, exactly like `/api/entity-atlas/import`) — body `{ json }` **or** `{ code }`. Handler `ApplyPresetImport(string body)`:
  1. **Size cap** — reject bodies > 2 MB (the shared `ReadBody` has no cap; add a checked read or length guard for this endpoint).
  2. If `code` present, `PresetCodec.TryDecode` → json; else use `json`.
  3. `JsonDocument.Parse` (System.Text.Json ignores unknown fields → forward-compatible); reject `Schema > 1`.
  4. **Auto-backup** the current look first: write the current `PresetBundle` to `config/presets/backup-before-import.poe2preset` (via `JsonStore.AtomicWrite`) so an import is recoverable.
  5. **Sanitize via the existing helpers** — `TryParseStyles`/`TryParseHpBars`/`TryParseTerrain`/`TryParseGroundItems` (expose them `internal static`) for the sub-objects; `SanitizeRule` on **every** imported `DisplayRule`.
  6. Apply: assign the sanitized sub-objects + bools onto `_settings`; if `DisplayRules` non-empty, `_displayRules.Replace(sanitizedRules)` (**destructive full swap** — see B.5; the `Generation` bump auto-propagates the live re-color to both threads); then `_settings.Save()`.
  7. Return `{ ok = true, rules = _displayRules.All.Count }`.

### B.4 Dashboard UI (Presets section in Settings)
A new "Presets" card in the Settings tab. Mirror the entity-atlas Export/Import JS:
- **Export** — "Copy share-code" (fetch `/api/preset/export`, `navigator.clipboard.writeText(code)`, toast) + "Download .poe2preset" (Blob → anchor download of `json`, filename `radar-preset.json`).
- **Import** — a paste-code `<textarea>` + Apply button (`POST {code}`) and a file-picker (static hidden `<input type=file accept=".poe2preset,application/json">` + `FileReader` → `POST {json}`, mirroring `#eaImport`).
- Import shows a **confirm dialog** first: *"Applying a preset replaces your display rules and visual styles (a backup is saved first). Continue?"* On success, reload settings + show a toast.

### B.5 Replace vs merge (decision)
A preset is "here's my complete radar look", so import **REPLACES** the display rules + visual styles (full swap) — matching the existing `/api/display-rules` replace semantics. The destructiveness is mitigated by (a) the explicit confirm dialog and (b) the auto-backup written before applying. (Merge is out of scope — YAGNI; a user wanting to keep their rules just doesn't import.)

### B.6 Hardening (folded in)
`RadarSettings.Save()` currently uses `File.WriteAllText` (not atomic) — since preset import adds a new write path, migrate `Save()` to `JsonStore.AtomicWrite` (crash-safe, matches the other stores). Small, isolated, de-risks both import and normal saves.

### B.7 Preset risks
- Migration-guard exclusion is enforced structurally by `PresetBundle` being a separate class (never serialize `RadarSettings`).
- `_settings` is shared across threads but the established pattern is lock-free field-by-field assignment + `Save()`; the import follows it (sequential assignments, one `Save`). No new locks.
- Untrusted share-codes: size-capped, schema-checked, every field sanitized/clamped via existing validators, nothing executable.

## Architecture / boundaries
- New Core: `Audio/PureToneWav.cs`, `Config/PresetCodec.cs` (both pure, unit-tested).
- New Overlay: `Audio/AudioCue.cs`, `Config/PresetBundle.cs`; additions to `RadarApp` (cue fields + `CheckAudioEvents` + dedup sets), `RadarSettings` (audio bools/int + atomic Save), `ApiServer` (audio-test + preset endpoints + `internal static` sanitizers), `DashboardHtml` (audio card + presets card).
- No Core→Vortice dependency; `Program.cs` console title untouched; all writes loopback-gated.

## Testing
- **Core unit tests (the test project references Core only):** `PureToneWav` (header/sample-count/amplitude/silence) and `PresetCodec` (round-trip, garbage rejection, prefix tolerance).
- **Build + review (Overlay):** audio event wiring, settings round-trip, dashboard cards, preset endpoints/sanitization (no Overlay test harness) — verified by build 0W/0E + full suite + compliance/scrub gates + the per-task and final reviews. Manual in-game smoke (audio fires on the three events; preset export→import round-trips a look) deferred to the validation session.

## Out of scope (YAGNI)
- TTS / voice; volume slider (fixed tone volume is fine for v1); a 4th "exit behind you" cue (noisier — drop).
- Preset MERGE; a preset gallery/registry; uploading presets to a remote server (file + code sharing is enough).
- Any change to what memory is read.
