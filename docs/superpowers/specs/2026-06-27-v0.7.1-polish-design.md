# v0.7.1 — Patch-Resilience Polish Bundle — Design

**Date:** 2026-06-27
**Branch:** `feat/v0.7.1-polish`
**Status:** Approved design — ready for implementation plan

## Goal

A small, solo, fast release of 6 mostly-independent polish items that harden the just-shipped v0.6.0 (patch-resilience) + v0.7.0 (Campaign GPS). Strictly read-only, compliance-clean. Ships as **v0.7.1**.

## Items

### 1. Force re-scan (the escape hatch)

When a patch breaks the offsets (or the auto-resolve picks a stale slot), a user stuck on the amber/red banner currently has no recourse but to restart. Add a manual force-re-scan:

- **`RadarApp.RequestRescan()`** — sets the volatile `_resolvedSlot = 0`. The existing `ResolverLoop` (v0.6.0) re-scans within its 1.5 s cadence; its re-attach branch already handles a dead process. No new threading.
- **`POST /api/rescan`** — a new ctor delegate `Action? rescan = null` on `ApiServer`, wired to `RadarApp.RequestRescan`; loopback-host-gated (same guard as `/api/nav` / `/api/settings`); returns `{ ok = true }`. GET/other → 405.
- **Dashboard** — a "Force re-scan" button inside the existing Status card (`#statusCard`), shown only when `healthState !== 'ok'`; on click it `POST`s `/api/rescan` and shows a brief "re-scanning…" acknowledgement.

### 2. Health pill in the masthead

Today game-health (attached / in-zone / reading-player) is only visible on the Settings tab's Status card. Surface it on **every** tab:

- A new element in `<header>` (next to the existing `#conn` chip): a colored dot + short label driven by `/state` `healthState` each tick — **green** = `ok`, **amber** = `searching`/`loading` (connecting), **red** = `broken`, **grey** = `waiting`/`notingame` (benign — game not running / at a menu); `healthMessage` as the `title` tooltip.
- `#conn` is unchanged (it's the dashboard↔overlay poll liveness); the new pill is the *game* read-health. Both live in the header.

### 3. Console polish

- Add the controller bindings to `ConsoleTheme.HotkeyList`: `R3` next target · `L3` prev · hold R3/L3 = fast-cycle · `L3+R3` = nav menu. (Keyboard rows already exist.)
- Make the version line in `ConsoleTheme.Banner()` prominent (its own gold line) rather than a dim suffix.
- Free branding fix: the dashboard masthead `<h1>OVERLAY</h1>` → `POE2GPS`. This is the loopback-only dashboard page, **not** the anti-detection console window title (which stays neutral, set in `Program.cs`), so the rename is safe.

### 4. WorldLoop error-hardening

A persistently-throwing `WorldTick` currently spams stderr ~30×/s and leaves the last (stale) world snapshot on screen. In `WorldLoop`'s `catch`:

- Call `PublishEmptyWorld()` so a thrown tick clears the radar instead of freezing stale data.
- Rate-limit the stderr log to ~once per 5 s (a simple `DateTime` gate field), so a hot-looping error doesn't flood the console.

(From `docs/audit-2026-06-22.md` §Correctness hardening.)

### 5. Dead/INVALID-offset cleanup (conservative)

`Poe2Offsets.cs` carries explicitly-`⚠ INVALID` and unreferenced GH2-port stubs that are copy-paste traps for future offset work. Delete **only** entries that are BOTH (a) marked `⚠ INVALID` or known-dead (`reads 0 — TBD`) AND (b) have **zero references in `src/`** (excluding the declaration itself — grep-verify each before deleting). Candidates: `ChestComponent` locked/large/destroys flags, `MonsterComponent.IsBoss`, `Targetable.*`, `PathfindingComponent.*`, `AreaTransitionComponent.*`, `InGameState.WorldData` + `UiRootStructPtr`, `ImportantUi`/`MapParent`, `Inventory.ServerRequestCounter`.

**Keep** documented "candidate" / speculative-but-not-wrong offsets that are future leads (e.g. `Render.ModelBounds`) — annotate, don't delete. The build (`TreatWarningsAsErrors`) + compliance gate are the safety net; any accidental deletion of a used symbol fails the build.

### 6. CHANGELOG + release checklist

- Create `CHANGELOG.md` at the repo root (Keep-a-Changelog style): one section per release v0.1.0 → v0.7.0 (+ an Unreleased/v0.7.1 stub), each with the headline features — sourced from the git tags + the project memory's per-release notes. Going forward, each release adds a section (a line in `docs/release-checklist.md` will remind).
- Add to `docs/release-checklist.md`: a `- [ ] Bump the "supports PoE2 X.Y.Z" badge in README.md` step (the CLAUDE.md rule that currently isn't in the checklist), and a `- [ ] Add the release to CHANGELOG.md` step.

## Architecture / boundaries

These are 6 independent, small changes — no shared new subsystem. Each touches a distinct surface:
- #1: `RadarApp` (a 1-line method) + `ApiServer` (ctor param + route) + `DashboardHtml` (button).
- #2: `DashboardHtml` (header element + tick render).
- #3: `ConsoleTheme` (HotkeyList + Banner) + `DashboardHtml` (h1).
- #4: `RadarApp.WorldLoop` (the catch block).
- #5: `Poe2Offsets.cs` (deletions only).
- #6: `CHANGELOG.md` (new) + `docs/release-checklist.md`.

## Error handling / safety

- `/api/rescan` is loopback-gated; `RequestRescan` only writes a volatile field the resolver already reads (no new races).
- The offset cleanup is deletion-only with grep-verify-before-delete + the build as backstop.
- WorldLoop hardening makes the existing catch *more* defensive, never less.

## Testing

Mostly wiring / cleanup / docs with no meaningful unit-test surface (the project tests Core only; these touch Overlay/data/docs):
- Verification = `dotnet build` 0W/0E + full xUnit suite green + `compliance-gate.ps1` + `scrub-strings.ps1 -SelfTest` green.
- Manual smoke: the re-scan button forces a reconnect; the masthead pill tracks health; the console shows controller hotkeys; a forced WorldTick exception clears the radar + logs once.
- The WorldLoop rate-limiter is a 3-line inline `DateTime` gate — not extracted, not unit-tested (YAGNI).

## Compliance

All additions are reads / UI / docs. `/api/rescan` sends no input and touches no game memory (it only resets a slot field). No forbidden symbols; the offset cleanup only *removes* code. README badge stays 0.5.4 (offsets unchanged).

## Out of scope (YAGNI)

- Any change to the resolver's scan cadence or re-attach logic (v0.6.0 already handles it).
- Auto-rescan heuristics — the button is manual on purpose.
- Deleting speculative "candidate" offsets that aren't marked INVALID (kept as leads).
- A full Keep-a-Changelog tooling pipeline — a hand-maintained file is enough.
