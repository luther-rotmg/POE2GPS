# v0.40 Arc Design — "Advanced Analytics"

**Status:** Approved by LO ("go"), 2026-07-17.

**Parent:** [2026-07-15-gold-tier-4-release-arc-design.md](2026-07-15-gold-tier-4-release-arc-design.md) — the 4-release arc named `Cartographer (movement heatmap + route replay)` as **v0.40 flagship** with the "strongest technical moat: uses the position stream nobody else has."

## Reader thesis

> "Every zone you cleared, drawn."

## The problem

POE2GPS reads player position (`Poe2Live.PlayerGrid(localPlayer) → Vector2`) every render tick (~30 Hz) but throws it away after using it for the current-frame center-of-radar calculation. The stream is dense, per-zone, per-character, and nobody else in the ecosystem has it — it's the tool's strongest untapped asset.

Two adjacent user needs both fit this stream:
- **Cartographer** (heatmap): "which parts of this zone have I walked, which are still unexplored?" A translucent heatmap layer over the map view, coloring visited cells with density.
- **Route replay**: "how did I run that map — which side rooms did I miss, where did I backtrack?" A scrubbable dotted-line playback of the position stream over the map.

Both consume the same underlying capture. Ship one storage layer, two visualizations.

## The proposal

**One position-capture pipeline** feeding **two rendering surfaces**:
- Storage: append-only per-character per-zone position log (`config/tracks/<character>/<zone>.jsonl`)
- Heatmap visualization: overlay/dashboard chip that renders a 2D density grid onto the current zone's map background
- Route-replay visualization: dashboard tab with a session list + scrubbable timeline that redraws position history as an animated path

### Locked architectural decisions (READ BEFORE ANY BEAD DISPATCH)

1. **Capture cadence** — sample position at 1 Hz (not per-tick). Downsample from render's 30 Hz. Rationale: heatmap resolution doesn't benefit past ~1 Hz, and 30 Hz would balloon on-disk storage 30× for near-zero visual gain.
2. **Storage shape**: `config/tracks/<sanitized-character>/<zone-code>.jsonl` — one JSONL line per position sample: `{t: <ms-since-zone-entry>, x: <grid-x>, y: <grid-y>}`. Character-name sanitize + zone-code stability gate same as v0.37 Codex.
3. **Zone lifetime**: track file opens on zone entry, appends per 1-Hz sample, closes on zone exit. Multiple sessions in the same zone APPEND to the same file (build up density over time). No per-session split — that's a v0.40.1 nicety if requested.
4. **Rendering resolution**: heatmap uses a 64×64 fixed-cell grid over the zone's bounds. Density = log(1 + samples-in-cell), normalized to [0, 1], then mapped through a viridis-ish color ramp (transparent-to-warm).
5. **Route-replay scrub**: playback speed presets 1×/4×/16×/max; timeline slider scrubs sample index. Dashboard tab NOT overlay — reserves overlay screen space.
6. **Storage cap**: 10K samples per (character, zone) file → ~150KB/file worst case. Beyond cap, drop oldest samples (ring). LO can raise later if needed.
7. **Endpoint contract**: `GET /api/tracks?character=<name>&zone=<code>` (loopback-gated because character-name leaks past machine — same rationale as `/api/codex`). Returns `{ samples: [ {t, x, y} ] }`.
8. **Privacy invariant**: character name NEVER appears in `/state` responses (already the case; must be preserved).

## Global constraints

- Read-only preserved (position is already read; no new memory-reach)
- No telemetry (config-local storage only)
- Additive (heatmap + replay both opt-in via new tab; existing overlay unchanged)
- Local-first (`config/tracks/`)
- Character-name privacy invariant preserved (`/state` never leaks name; only loopback-gated `/api/tracks`)

## Known risks + mitigations

1. **Heatmap perf regression at high sample counts** — 10K samples × 64 grid cells = trivial. Bench: single reduce pass over samples building a `Dictionary<(int,int),int>` should be <1ms even at 100K samples. Ship bench test in P1.
2. **Zone-code stability during login flicker** — reuse v0.37 SessionEventLog's 30-tick stability gate.
3. **Character-name typo/rename mid-session** — file-name sanitization + new-name → new file. Old file survives untouched.
4. **Multi-session append clobbering** — file lock on append + `.tmp` staging fallback if lock fails. Documented in P1 spec.
5. **Storage growth** — 10K/file × ~20 unique zones × 3 characters = ~10MB/character total worst case. Acceptable local storage; no rotation needed at this scale.

## What v0.40 ships (release notes shape)

**"Cartographer — every zone you cleared, drawn."**

- 🗺 **Movement heatmap.** New Cartographer tab shows which parts of each zone you've walked, colored by density. The tool has been sampling your position at 1 Hz behind the scenes; run the same map twice and the coverage grows.
- ▶️ **Route replay.** Timeline scrubber + play/pause plays back your movement through any zone you've cleared. See where you backtracked, which side rooms you missed. 1×/4×/16×/max playback speeds.
- 🔒 **Loopback-gated `/api/tracks`.** Character-name query is only served to localhost — same privacy posture as v0.37's Codex endpoint. Position data never leaves your machine.
- 🎯 **Additive.** No overlay changes. Ignore the Cartographer tab and nothing about your rig changes. The 1-Hz sampler is cheap enough to leave running.

## Post-arc backlog (v0.40.x)

- Per-session split (currently multiple runs of same zone APPEND to same file — v0.40.1 could offer split-by-session view)
- Export as PNG / GIF for streaming clip use
- Cross-zone aggregate ("all your Delirium mirror clears overlaid")
- Density-based route suggestion ("shortest path connecting all unvisited cells")

## Open questions still needing LO awareness

1. **Storage cap** — 10K samples/file default. LO can raise via `config/tracks-cap.json` future config.
2. **Sample rate** — 1 Hz sample rate hardcoded in v1.0. Configurable via `config/tracks-hz.json` in v0.40.1 if user requests.
3. **Overlay integration** — v1.0 dashboard-only. If overlay heatmap layer (translucent grid drawn atop the current-zone radar) is desired, that's v0.41 scope.
