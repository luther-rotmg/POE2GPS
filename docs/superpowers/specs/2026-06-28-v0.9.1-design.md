# v0.9.1 — Audio polish + Preset library + Zone summary — Design

**Date:** 2026-06-28
**Branch:** `feat/v0.9.1`
**Status:** Approved direction — spec for implementation

## Goal

A QoL release that completes and polishes the v0.9.0 features, plus one new opt-in panel. Three cohesive areas, all strictly read-only / GGG-compliant, all solo-buildable (no memory RE):
- **Audio polish** — a master volume slider + a per-event tone picker for the alert cues.
- **Preset library** — built-in curated starter presets + save/name/apply/delete your own, turning the Presets card into a real library.
- **Zone summary panel** — a small opt-in overlay panel with live counts for the current zone.

The `Ctrl+Alt+R` console-banner fix already merged to `main` rides along in this release. README/CHANGELOG/version updated at the end.

## Compliance invariant

Volume + tones are audio **output** (no input/injection/memory-writes). The preset library is loopback-gated config I/O. The zone summary just aggregates entity data the radar already reads. No new game-process interaction. Compliance gate + scrub stay green; README badge stays `0.5.4`.

## Area 1 — Audio polish

### 1.1 Tone table (Core, testable)
A small named-tone table in `POE2Radar.Core/Audio/ToneTable.cs`: a static map of name → `(double FreqHz, int DurationMs)`, all rendered through the existing `PureToneWav.Generate` (sine + envelope). Set:
`Chime (660,120)`, `Bell (880,150)`, `Ding (520,180)`, `Beep (880,90)`, `Blip (440,70)`, `Alert (1040,110)`, `Low (330,160)`.
API: `ToneTable.Names` (ordered list for the UI), `ToneTable.TryGet(name, out freq, out ms)` with a safe fallback to a default tone for unknown names. **Unit-tested** (every named tone resolves; unknown name falls back; each produces a valid non-silent WAV via `PureToneWav`).

### 1.2 Settings
`RadarSettings` gains four flat auto-properties (round-trip via the camelCase serializer like the existing audio keys):
- `AudioAlertVolume` (int, default **70**) — 0–100, clamped on apply.
- `AudioToneMonster` (string, default `"Chime"`), `AudioToneItem` (default `"Bell"`), `AudioToneObjective` (default `"Ding"`) — the defaults reproduce today's 660/880/520 Hz cues exactly.

### 1.3 Cue rebuild
`AudioCue` already pre-loads one WAV. Add `RadarApp.RebuildAudioCues()` that reads the three tone settings + volume and recreates the three `AudioCue`s via `ToneTable` + `PureToneWav.Generate(freq, ms, AudioAlertVolume/100.0)`. Called once in the ctor (replacing the hardcoded `Generate` calls) and whenever `/api/settings` applies any of the four audio-output keys — wired via a new `ApiServer` ctor delegate `Action? rebuildAudio` (mirroring the existing `rescan`/`audioTest` delegates), invoked from the settings POST handler when a volume/tone key changes. The three cue fields become reassignable (drop `readonly`); replacing the reference is safe (old `SoundPlayer` is released; never `Dispose` mid-play — just drop it).

### 1.4 Dashboard (Audio card)
The existing Audio alerts card gains: a **volume** `<input type="range" min=0 max=100 data-set="audioAlertVolume">` and three tone `<select data-set="audioToneMonster">…` populated from `GET /api/audio/tones` (returns `ToneTable.Names`) — or simply hardcode the option list in the HTML to match `ToneTable.Names` (simpler; one source of truth is the table, the HTML mirrors it). The existing **Test** buttons audition the current pick (they already call `/api/audio-test`, which plays the current cue — which is now the rebuilt one). Numeric/select `data-set` auto-binding already exists.

## Area 2 — Preset library

### 2.1 Name sanitizer (Core, testable)
`POE2Radar.Core/Config/PresetName.cs`: `static string Sanitize(string raw)` → a safe filename stem. Whitelist `[A-Za-z0-9 _-]`, collapse/trim whitespace, strip everything else (so `..`, `/`, `\`, `:` etc. are removed — **no path traversal**), cap length (e.g. 48), and return a fallback (`"preset"`) if the result is empty. **Unit-tested** (path-traversal attempts → safe stem; unicode/punctuation stripped; empty → fallback; length cap; idempotent).

### 2.2 PresetStore (Overlay)
`POE2Radar.Overlay/Web/PresetStore.cs` — manages user presets under `config/presets/*.poe2preset` plus the embedded built-ins:
- `IReadOnlyList<(string Name, bool BuiltIn)> List()` — built-ins first (⭐), then local files (by name).
- `string? Load(string name)` — returns the bundle JSON (built-in resource or local file); null if absent.
- `void Save(string name, string bundleJson)` — `PresetName.Sanitize` → `JsonStore.AtomicWrite` to `config/presets/<safe>.poe2preset`.
- `bool Delete(string name)` — deletes a **local** file only (built-ins are immutable); returns false for a built-in or missing name.
Reserved-name guard: a local save that sanitizes to a built-in's name is suffixed (e.g. `-1`) so it can't shadow a built-in.

### 2.3 Built-in starter presets (embedded)
Three curated `PresetBundle` JSONs embedded as resources (`Web/presets/high-contrast.poe2preset`, `minimal.poe2preset`, `boss-hunter.poe2preset`):
- **High-contrast** — brighter rule colors, larger icon sizes, thicker/visible HP bars (max legibility).
- **Minimal** — only the essentials enabled (monsters by rarity, chests, transitions); muted colors, small icons; ground-item labels off.
- **Boss & unique hunter** — emphasize Rare/Unique monsters + bosses (large, bold), dim Normal/Magic; uniques on the ground highlighted.
Each is a valid `PresetBundle` (Schema 1) built from the current defaults with the above tweaks; authored + sanity-checked during implementation so `ApplyPresetImport` accepts them cleanly.

### 2.4 Endpoints
Mirroring the existing preset endpoints:
- `GET /api/preset/list` (read-only, no gate) → `[{ name, builtIn }]`.
- `POST /api/preset/save` `{ name }` (POST + loopback) → build the current `PresetBundle` (same as `/api/preset/export`) → `PresetStore.Save`.
- `POST /api/preset/apply` `{ name }` (POST + loopback) → `PresetStore.Load(name)` → existing `ApplyPresetImport(bundleJson)` (auto-backup + full sanitize + Generation bump — unchanged path).
- `POST /api/preset/delete` `{ name }` (POST + loopback) → `PresetStore.Delete`.
The existing `GET /api/preset/export` and `POST /api/preset/import` (code/file) stay.

### 2.5 Dashboard (Presets card)
The Presets card gains a **library list**: each entry shows the name (built-ins marked ⭐) + an **Apply** button; local entries also get **Delete**. Above/below it: a **"Save current as …"** text input + Save button. Apply uses the confirm-dialog (same as import). The existing copy-code / download / paste / file controls remain.

## Area 3 — Zone summary panel (opt-in)

### 3.1 Behavior
A small overlay panel (mirroring the existing Monolith panel) showing live counts for the current zone:
- **Rares** — alive monsters with `Rarity == Rare`.
- **Uniques** — alive monsters with `Rarity == Unique`.
- **Chests** — `Category == Chest` not yet opened.
- **Exits** — transition landmarks/entities in the zone (labelled "exits"; a simple count of `Transition`-category targets — not a true explored/unexplored diff, which would need pathing state; keep it a straight count for v1).
Counts are computed on the world thread (cheap aggregate over `_entities` + landmarks) and published in the world snapshot; the renderer draws them. Off by default.

### 3.2 Settings + PROMINENT toggle
`RadarSettings.ShowZoneSummary` (bool, default **false**) + `ZoneSummaryCorner` (string, default `"TopRight"`, reusing the Session-HUD corner convention).
**The toggle must be prominent** (user requirement — it's opt-in): a dedicated **"Zone summary"** card placed **near the top of the Settings tab, immediately after the Audio alerts card** (the two opt-in QoL features sit together at the top, above the Target-cycling card). The card has the on/off switch + a corner selector.

### 3.3 Rendering
`OverlayRenderer.DrawZoneSummary(rt, ctx)` mirroring `DrawMonolithPanel`: a compact titled panel ("Zone") with the four labelled counts, anchored to `ZoneSummaryCorner`, drawn only when `ShowZoneSummary` and in-game. `RenderContext` carries the four counts (a small `ZoneSummary` value struct) published from the world tick.

## Architecture / boundaries
- New Core (pure, tested): `Audio/ToneTable.cs`, `Config/PresetName.cs`.
- New Overlay: `Web/PresetStore.cs`, three embedded `Web/presets/*.poe2preset`, `RadarApp.RebuildAudioCues`, `OverlayRenderer.DrawZoneSummary`; additions to `RadarSettings` (4 audio + 2 zone-summary keys), `ApiServer` (audio-rebuild delegate + tones/list/save/apply/delete endpoints), `DashboardHtml` (audio volume/tone controls + preset library list + zone-summary card), `RenderContext` (ZoneSummary struct).
- No Core→Vortice; `Program.cs` console title untouched; all writes loopback-gated; preset apply keeps the existing sanitize+backup path.

## Testing
- **Core unit tests:** `ToneTable` (resolve/fallback/valid-WAV) and `PresetName.Sanitize` (path-traversal, strip, fallback, length, idempotent). Both pure → fit the Core-only test project.
- **Build + review (Overlay):** cue rebuild, preset store/endpoints/UI, zone-summary aggregation + render — no Overlay harness; behavior-preserving wiring verified by build 0W/0E + full suite + compliance/scrub gates + per-task and final reviews. In-game smoke (tones change, volume audible, presets apply, panel shows correct counts) deferred to the validation session.

## Out of scope (YAGNI)
- Multi-segment / two-tone cues (single-tone variants only this release).
- Per-cue volume (single master volume).
- Preset thumbnails/preview, a remote preset gallery.
- True explored/unexplored exit tracking (a straight transition count for v1).
