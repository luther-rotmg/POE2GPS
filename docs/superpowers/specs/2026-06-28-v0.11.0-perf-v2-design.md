# v0.11.0 — Performance v2 (Bucket-2 syscall/alloc reductions) — Design

**Date:** 2026-06-28
**Branch:** `feat/v0.11.0-perf-v2`
**Status:** Approved direction — spec for implementation

## Goal

Land the 8 deferred **Bucket-2** memory-read optimizations from `docs/audit-2026-06-27-perf.md`
as one behavior-preserving release, plus a small always-on RPM counter so the footprint win is
**measured, not estimated**. Bucket 1 (the SOLO, offline-validatable half) shipped in v0.8.0; these 8
were deferred because they change *how* memory is read and need a live entity/map view to confirm
every value still resolves.

For this project, "footprint" = (a) ReadProcessMemory syscalls/sec, (b) per-tick managed allocations /
GC pressure. These 8 target the largest raw-syscall reductions on the world hot path.

## Compliance invariant (non-negotiable)

Everything here is a **read-side** refactor. Strictly read-only: `OpenProcess` read-only +
`ReadProcessMemory` only; **no** new offsets; **no** input/process-write APIs. The RPM counter only
*counts* existing reads. No Core→Vortice dependency. `scripts/compliance-gate.ps1` +
`scripts/scrub-strings.ps1 -SelfTest` stay green. `Program.cs` console title untouched. README badge
stays `0.5.4`.

**Behavior-preserving is the bar:** entities, rarity, hostility coloring, HP bars, the map element
(shift/zoom/visibility), item mods, terrain, and player vitals must be **identical** before/after each
change. Anything that changes the *drawn* result is a regression, not an optimization.

## Part 0 — Measurement harness (build first)

### 0.1 RPM counter
A single `Interlocked.Increment` on the one RPM chokepoint (`ProcessHandle.ReadProcessMemory` /
`MemoryReader`). Expose a cumulative `long RpmCalls` (read via `Volatile.Read`). Cost is negligible
(one atomic add per read) and it is always on — no debug flag.

### 0.2 Surface as a rate in `/state`
`Web/ApiServer.cs` tracks `(lastRpmCalls, lastRpmAtTicks)` and, on each `/state` poll, computes
`rpmPerSec = (RpmCalls - lastRpmCalls) / elapsedSeconds`, then updates the snapshot. `/state` gains a
numeric `rpmPerSec` field alongside the existing `worldMs` / `renderMs`. (Dashboard surfacing of the
number is optional/nice-to-have, not required for the release.)

### 0.3 Baseline
**Before any optimization**, run the overlay on a busy (~300–400 monster) map and record `rpmPerSec`,
`worldMs`, and `renderMs` from `/state`. This is the "before" column for the release notes. Re-read
after each tier to watch the number fall.

## Part 1 — Tier 1: pure caches / mechanical (lowest risk, ship first)

### 1.1 ReadChestOpened one-way cache (`Poe2Live.cs:1375-1380`)
The opened flag never reverts. Add `_openedChests` (`HashSet<nint>`), cleared in the existing
per-area cache invalidation (cache key = AreaInstance address). In `ReadChestOpened(addr)`: if `addr`
is in the set, return `true` without reading; otherwise read, and if opened, add `addr` to the set.

### 1.2 PlayerVitals world-thread split (`RadarApp.cs:1157`, `Poe2Live.cs`)
The world thread calls `_live.PlayerVitals(...)` only for the offset-heal side-effect (it latches the
vital offsets on first valid read) but reads and discards HP/Mana/ES every tick. Expose
`Poe2Live.EnsureVitalOffsets()` that runs *only* the latch logic and returns nothing; `WorldTick`
calls that instead. The render thread is unchanged — it still reads full vitals via `_liveRender` for
HP bars. (The existing offset-re-latch correctness guard from v0.8.0 is preserved inside the latch
logic.)

### 1.3 TryReadMapElement single bulk read (`Poe2Live.cs:1057-1065`)
Replace the 4 separate per-frame reads (Flags `+0x180`, Shift `+0x368`, DefaultShift `+0x370`, Zoom
`+0x3A8`) with one `TryReadBytes` over the contiguous range `[+0x180 .. +0x3AC)` (~0x22C bytes),
slicing the four fields from the buffer in-process. ~4 RPM/element/frame → 1 (~1.2k syscalls/s at
144 Hz). Plan re-confirms the four offsets are contiguous and the whole range is readable.

### 1.4 Terrain raw-buffer ArrayPool (`Poe2Live.cs:958`)
The intermediate packed-grid `byte[]` is one LOH allocation per zone change. Rent it from
`ArrayPool<byte>.Shared` (size ≥ needed), use it, and `Return` in a `finally`. The terrain bitmap bake
already copies out what it needs, so no rented buffer is retained past the method.

## Part 2 — Tier 2: big bulk-reads (the headline wins)

### 2.1 ResolveComponent bulk-read (`Poe2Live.cs:1442-1468`) — the #1 item
Today each needed component triggers its own walk of the ComponentLookUp bucket (~9 needed components
→ ~9 full walks), allocating a string per entry compared (~50–140 string allocs/entity). Replace with
a **single pass**: read the bucket array once, iterate in-process, and resolve *all* needed component
addresses in that one pass by comparing entry name **bytes** against the known component-name byte
patterns (no per-comparison string allocation). Net: ~9 walks → ~1 per new entity, string allocs
eliminated. This is the highest per-pack win and the change most worth the live parity check.

### 2.2 ReadMods bulk-read (`Poe2Live.cs:614-628`)
Bulk-read the mod array (`ModArrayStruct` stride `0x40`, length = mod count) in one RPM, iterate
in-process, and dedup mod ids with a `HashSet`. Drives the mod catalog / item labels.

## Part 3 — Tier 3: nuanced, with explicit guards (ship last, heaviest validation)

### 3.1 ReadReaction value cache + slow refresh (`Poe2Live.cs:728-737`)
Hostility (Reaction) is static per spawn in the common case, so cache it in a `_reaction`
`Dictionary<nint,byte>` mirroring `_rarity` (cleared on area change). **Guard for the conversion
case** (enemy↔friendly, e.g. dominating-blow / certain league mobs): refresh the whole reaction cache
every `ReactionRefreshTicks` world ticks (default **30**, ≈ 1 s at 30 Hz) by clearing it so all
entities re-read that tick. Even re-reading 400 entries once/second is ~13 reads/tick amortized vs
400/tick — the ~12k RPM/s win survives the guard, and any conversion resolves within ~1 s. The cadence
constant lives in one place (a `const`), documented.

**Testable seam:** the "should the reaction cache refresh on this tick?" decision is a pure function
`(tickCounter, refreshInterval) -> bool` — unit-tested in the Core project.

### 3.2 Entities() list reuse via triple-buffer (`Poe2Live.cs:415`)
Today `Entities()` does `new List<EntityDot>(256)` per world tick; the filled list is published in the
immutable `WorldSnapshot` and read lock-free by the render thread. Naively reusing one list would let
the render thread read a list while the world thread clears+refills it (torn read). Guard with a
**triple-buffer**: keep `List<EntityDot>[3]`, advance an index each call, `Clear()` + fill the current
buffer, and publish it. With 3 buffers a just-published list is not reused for 2 more world ticks
(~66 ms), far beyond render read latency, so the render thread never reads a buffer being refilled.
(The simpler capacity-hint variant already shipped in v0.8.0; this removes the allocation entirely.)

**Escape hatch:** #7 is the only change with genuine concurrency risk. If live soak shows *any* entity
flicker / torn list, revert this one change and ship the other 7 — the spec explicitly permits
dropping 3.2 without re-approval.

## Validation strategy (per change, live)

Each change: build 0W/0E + full Core test suite + compliance/scrub gates. Then, for the read-pattern
changes, a ~30-second live parity check using the one-click probe kit, confirming identical results
and watching `rpmPerSec` / `worldMs` fall:

| Change | Live parity check |
|---|---|
| 2.1 ResolveComponent, 3.1 ReadReaction | `--rarity` / `--entity` parity; hostility colors correct on a busy map; (3.1) a conversion scenario if available |
| 2.2 ReadMods | `--inventory --itemmods` parity |
| 1.3 Map element | minimap shift/zoom/visibility track correctly while moving + zooming |
| 1.1 Chest, 1.4 Terrain, 1.2 Vitals | chests flip to open and stay; terrain redraws clean across 3-4 zones; HP bars match orbs |
| 3.2 Entities reuse | soak a 300+ monster map several minutes — zero flicker / torn entity lists |

Record `rpmPerSec` + `worldMs` before the batch and after each tier; the before/after numbers go in
the CHANGELOG and the Discord post.

## Architecture / boundaries

- **Core:** `Game/Poe2Live.cs` (1.1, 1.3, 1.4, 2.1, 2.2, 3.1, 3.2 + `EnsureVitalOffsets` for 1.2);
  `ProcessHandle.cs` / `MemoryReader.cs` (RPM counter). No new offsets; `Game/Poe2Offsets.cs`
  unchanged.
- **Overlay:** `RadarApp.cs` (1.2 call `EnsureVitalOffsets`; nothing else behavioral);
  `Web/ApiServer.cs` (`rpmPerSec` in `/state`).
- No Core→Vortice. All API stays loopback-gated and read-only.

## Testing

- **Core unit tests:** the chest-cache add/skip/clear semantics and the reaction-refresh cadence
  predicate where they factor into pure functions; both fit the Core-only test project.
- **Live-validated (no Overlay harness):** every read-pattern change is behavior-preserving and
  verified by the parity table above + build 0W/0E + full suite + compliance/scrub gates + per-task &
  final review.

## Out of scope (YAGNI)

- New offsets or new memory-RE (this is pure read-pattern refactoring).
- Any change to *what* is drawn, ranked, or detected.
- Dashboard UI beyond optionally surfacing `rpmPerSec`.
- Bucket-1 / hygiene items already shipped in v0.8.0, and anything not in the 8 listed here.
