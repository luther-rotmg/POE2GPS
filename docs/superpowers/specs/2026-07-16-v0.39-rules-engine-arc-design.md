## Reader thesis

> "One place, one schema, one Save. Every kind of per-entity behavior lives here."

## The problem

POE2GPS has **five** independent rule surfaces today, each with its own storage shape, dashboard tab, and application seam:

| Surface | Storage | Dashboard tab | Applied at |
|---|---|---|---|
| **Display rules** (entity visibility/tint/priority) | `RadarSettings.DisplayRules` — flat list keyed by `Metadata` path | Settings → Display Rules | Renderer per-entity walk |
| **Hidden mobs** (allowlist entity-tokens to force-hide) | `RadarSettings.HiddenMobIds` — `HashSet<string>` | Settings → Hidden Mobs | ItemFilterEngine early-exit |
| **Landmarks** (custom POIs the user cataloged) | `config/custom-landmarks.json` — separate file, `CustomLandmarkData` records | Settings → Landmarks | Renderer overlay + minimap |
| **Affix nameplates** (item affix priority for the nameplate ribbon) | `RadarSettings.AffixNameplateRules` — regex-per-rarity | Settings → Affix Nameplates | ItemFilterEngine ribbon render |
| **Buff nameplates** (entity buff highlighting) | `RadarSettings.BuffNameplateRules` — buff-name→color map | Settings → Buff Nameplates | Renderer buff-icon strip |

Consequence: adding a NEW behavior ("boss encounters get a red aura", "delirium mobs pulse", "notable X gets custom label") requires either shoving it into one of these fixed surfaces or authoring a sixth. Copy-paste rule authoring across tabs is common (per-metadata predicate looks similar in 3 of 5 surfaces). Users can't compose rules ("hide, UNLESS unique-rarity, UNLESS in-hideout").

## The proposal

**One unified rule schema** with a **selector → effect** shape, layered on TOP of the five legacy surfaces (not a rewrite). Legacy surfaces continue to work byte-identically; the new engine is opt-in via a new dashboard tab and a per-rule `enabled` flag.

### Locked architectural decisions (READ BEFORE ANY BEAD DISPATCH)

1. **Rule shape**: `{schemaVersion:1, id, name, priority, enabled, when: <selector>, then: <effect list>}`. `id` is a UUIDv4 assigned at Save. `priority` is an integer, higher wins on conflict.
2. **Selector grammar (v1)** — a flat AND of predicates, no nesting yet:
    ```
    { metadata?: "regex", token?: "regex", rarity?: "unique|rare|magic|normal", zoneCode?: "regex",
      inHideout?: bool, minLevel?: int, maxLevel?: int, hasBuff?: "regex" }
    ```
    v1.1 can add `not:`, `any:`, `all:` nesting once shape stabilizes.
3. **Effects (v1)** — union list, each independently applicable:
    - `hide`
    - `tint: "#rrggbb"`
    - `ring: "#rrggbb"` (draws outline)
    - `label: "text with {tokens}"` (`{name}`, `{level}`, `{metadata}`, `{zone}`)
    - `sound: "wav-filename"` (files in `config/sounds/`, silent-fallback on missing)
    - `pulse: "slow|fast"`
4. **Storage**: `config/rules.json` — whole-file envelope `{schemaVersion:1, rules:[...]}`, atomic write via `.tmp` rename (existing pattern from RadarSettings). No JSONL for this one — rule set is small (10s not 1000s), rewrites are cheap, and full-file writes let us do trivial diff-review.
5. **Compilation**: `RuleEngine.Compile(RulesFile) → CompiledRuleSet` — regex pre-compiled, effects deduped, sorted by descending priority. Renderer + ItemFilterEngine both consume `CompiledRuleSet`. Compile at load + on Save; NEVER at per-entity tick.
6. **Legacy interop**: Legacy surfaces (DisplayRules / HiddenMobIds / AffixNameplateRules / BuffNameplateRules / CustomLandmarks) continue to work UNCHANGED. New engine layers ON TOP — its effects apply AFTER legacy has run, so a legacy `hide` is honored even if a new-engine `tint` matched. A new dashboard hint pointer explains "you can migrate rule X here" per-tab.
7. **UI: single new tab, no auto-migration**. Manual "Migrate this legacy rule →" button on legacy tabs copies the rule spec into the new engine (with a suggested `id` + `name`) and leaves the legacy source in place so the user can compare/verify before deleting.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Per-tick match cost for large rule sets | Compile once, pre-index by rarity+zoneCode for coarse filter, then run remaining predicates per entity. Cap at 100 rules with dashboard warning at 50+. |
| Regex authoring footguns (invalid regex crashes overlay) | Store validation at Save AND at Compile time; invalid rules skip with dashboard warning; NEVER crash renderer. |
| Legacy vs new-engine conflict confusion | Post-render tooltip: hover an entity in the overlay, popup shows "matched by: DisplayRule[3], Rule[boss-alert]" with priority ordering. Deferred to v0.39.1 as scope-guard. |
| Migration button correctness across 5 legacy shapes | Ship migration per surface as separate R7.x sub-tasks; users can migrate opt-in per rule; original untouched until they manually delete. |

## What v0.39 ships (release-notes shape)

**"Rules Engine v1 — one place for every per-entity override."**

- **New unified Rules tab.** Every rule is `when this matches, then do this`. Selector filters by metadata regex, entity token, rarity, zone code, in-hideout flag, level range, buff presence. Effects: hide, tint, ring outline, custom label with `{token}` templates, sound, pulse.
- **Composable priority + enabled toggle.** Higher priority wins conflicts; toggle any rule without deleting it. Save → apply live, no restart.
- **Legacy migration on demand.** Every existing rule tab (Display Rules, Hidden Mobs, Landmarks, Affix Nameplates, Buff Nameplates) gets a "Migrate this rule →" button that copies the spec into the new engine. Original stays until you delete it — nothing auto-changes.
- **Fully additive.** All existing legacy rule surfaces work byte-identically. The new engine is opt-in. Ignore the new tab and nothing changes about how the overlay behaves for you.
- Under the hood: `POE2Radar.Core.Rules.RuleEngine` with pre-compiled selectors, `config/rules.json` on-disk envelope, `/api/rules` loopback-gated CRUD.

## Second-slot options (open question for LO)

The arc-parent doc reserves the second v0.39 slot for **Streamer Kit** (Death-cam Replay + Channel Brand Kit). Given Rules Engine at 7 beads / ~1200 LOC is already medium-large, my recommendation:

- **A. Ship v0.39 as Rules-Engine-only** — 7 beads, one flagship, one release. Streamer Kit lands as **v0.39.1** or v0.40 flagship. Rationale: keeps beat sequencing crisp ("one theme per release"), matches v0.38's tight-arc pattern.
- **B. Ship Rules Engine + Death-cam Replay only** — skip Brand Kit (font-licensing gray area per arc doc), add 3-4 Death-cam beads. Requires the `/obs` cel endpoint + SVG rasterizer that arc doc flagged as MISSING. Risk: two subsystems in flight raises scope.
- **C. Ship Rules Engine + Stream-Safe polish** — extend v0.35's Stream-Safe with more masks (character-name overlay, level-up popup censor). Small (~200 LOC), doesn't need `/obs`. Fits the streamer beat without full Death-cam.

Recommendation: **A** for narrative clarity. B and C are viable but expand scope.
