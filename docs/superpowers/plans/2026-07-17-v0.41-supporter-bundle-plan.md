## 🚨 Locked Architectural Decisions (READ BEFORE ANY BEAD DISPATCH)

**1. `SupporterGate` API** (single source of truth, canonical everywhere):
```csharp
public static class SupporterGate {
    public static bool IsSupporter { get; }
    public static void RefreshFromSettings(RadarSettings settings);
}
```
JS mirror: `window.__supporterGate = { isSupporter: () => bool, refresh: async () => void }`. Refresh is idempotent; safe to call from any event handler. Retrofits the existing `applySupporterCosmetics` gate — that function calls `SupporterGate.refresh()` at its start.

**2. `ZoneCodeMatcher` API** (wildcard-prefix, glob syntax):
```csharp
public static class ZoneCodeMatcher {
    public static bool Match(string pattern, string zoneCode);
}
```
Glob syntax: `*` = wildcard anywhere; `*_town` matches `G1_town`; `T17_*` matches `T17_Necropolis`; `*` alone matches anything. Case-sensitive. NO regex escaping — treat `*` as the sole special char.

**3. `SupporterHint` component** (JS + CSS, inline card):
Renders in-place where a supporter feature editor would be, for non-supporters. Copy: `"Save your Ko-fi code in Settings to unlock this."` with a "Go to Settings →" link that switches the active tab. CSS class `.supporter-hint` reuses the `--gold` border theme from v0.38 Color Forge for visual consistency.

**4. Storage envelope** (all 4 flagship stores):
Whole-file `{schemaVersion: 1, ...}` envelope, atomic write via `.tmp` + `File.Move`, ArgumentException on validation failure. Mirrors v0.39 `RulesFileStore` from `RuleModel.cs`/`RulesFileStore.cs`. NEVER JSONL for these — none grow unbounded.

**5. Endpoint contract** (all 4 flagship endpoints):
- `GET /api/<feature>` — list, ungated
- `GET /api/<feature>/<id-or-key>` — single, ungated
- `POST /api/<feature>` — save, **loopback-Host-gated**
- `DELETE /api/<feature>/<id-or-key>` — loopback-Host-gated
- 409 on duplicate-name at create; 400 on validation; non-loopback → 403 `{error:"loopback-only"}`
Mirrors v0.38 `/api/palettes` + v0.39 `/api/rules` pattern.

**6. Radar Filter selection** (first-match-in-array-order wins):
Presets iterate in the order they appear in `radar-filters.json`. First `ZoneCodeMatcher.Match(preset.match, currentZone)` == true wins. Applies to both A (radar filter) and B (overlay layouts).

**7. Radar Filter caps**: 20 presets (dashboard warning at 15), 50 patterns per whitelist/blacklist list. Overlay Layouts cap: 10 layouts (warning at 8). Nav Destinations cap: 50 destinations (warning at 40).

**8. Nav Destinations use EXACT zoneCode equality** (intentional divergence from A + B). A named "chest room" location only makes sense for a specific zone; wildcards would produce misleading routes. `ZoneCodeMatcher` is NOT used for C.

**9. Session Widget avg-map-clear-time** — rolling average over the last 10 completed map clears in this session. If fewer than 3 completed, chip displays `"—"` (em-dash).

**10. NO worker git/bd operations. PRESERVE ALL OTHER BEHAVIOR. xUnit only (plain Assert.*). No AI attribution. No character name in `/state`.**

## Known risks + mitigations

1. **Perf claim measurability (A2)** — Blacklist perf gain must be observable. Ship xUnit bench that runs the renderer's entity-draw loop against a 100-entity synthetic snapshot, once with blacklist active + once without; assert active is at least 20% faster or ships a `Skip` reason. Sample bench pattern from v0.39 R2 (100 rules × 10K iterations).
2. **Panel `data-panel-id` audit (B2)** — Some overlay panels may not currently have the attribute. B2 audits and backfills; discovered panels documented in the bead output.
3. **Custom nav-destination A\* seam (C2)** — Existing landmark render path may not accept dynamic-injection cleanly. C2 scope-guard: if the injection point requires an iterator refactor, split into C2a (iterator refactor) + C2b (injection).
4. **Session widget drag-position (D3)** — Overlay panels don't have a shared drag framework yet (v0.41 "Dashboard Studio" per arc parent doc); D3 uses simple x/y config with position editor in dashboard (not overlay drag). Defers drag-to-position UI to a future v0.41.x.
5. **`SupporterGate` refresh timing (S1)** — Must refresh when user changes Ko-fi code without page reload. S1 subscribes to the existing settings-changed event flow (grep current `applySupporterCosmetics` for the pattern).
6. **Worker success rate on complex UI beads (A3/B4/C4)** — Empirical rate ~60% first-try on complex dashboard beads. Frontier-fix-in-place is the mitigation; do not re-dispatch unchanged. Budget for 1-2 salvage passes across the arc.
7. **Test-count budget** — 22 beads × ~10 xUnit facts each = ~220 new tests. Suite growing from 1112 → ~1330. CI wall time acceptable (~20s at current pace).
