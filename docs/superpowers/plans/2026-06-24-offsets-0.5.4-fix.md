# PoE2 0.5.4 Offset Hotfix Implementation Plan

> **For agentic workers:** Mechanical offset re-validation. Execute inline (the edits
> are integer-constant changes — a trivial mechanical edit, not a TDD feature). The
> verification cycle is **build + compliance gate + an exhaustive completeness-audit
> workflow + live in-game probes**, not unit tests — memory offsets have no CI-testable
> surface (they only validate against a running game). Spec:
> `docs/superpowers/specs/2026-06-24-offsets-0.5.4-fix-design.md`.

**Goal:** Restore the core read chain on PoE2 0.5.4 by applying the contained +0x18
`AreaInstance` shift, verify completeness, and ship as v0.5.1.

**Architecture:** Single source of truth is `Poe2Offsets.cs`; the live overlay
(`Poe2Live.cs`) already routes every read through `Poe2.AreaInstance.*` constants, so the
table edit fixes the overlay. Research probes that hardcode raw hex get routed through the
same constants. Docs + version follow.

**Tech Stack:** .NET 10 (net10.0-windows, x64), C#, `POE2Radar.slnx`.

## Global Constraints

- **Read-only / compliance:** integer-constant + comment edits ONLY. No new memory writes,
  no input/automation, no pricing/market data. `scripts/compliance-gate.ps1` and
  `scripts/scrub-strings.ps1 -SelfTest` must pass.
- **Exact values:** the 5 new offsets are `ServerDataPtr=0x598`, `LocalPlayer=0x5B8`,
  `AwakeEntities=0x6D8`, `SleepingEntities=0x6E8`, `TerrainMetadata=0x8B8`. Verbatim.
- `PlayerServerDataVec` **stays 0x48** — do not change it.
- Build clean: `dotnet build POE2Radar.slnx -c Release` → 0 errors, 0 warnings.
