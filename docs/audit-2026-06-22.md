# Overnight audit — findings & triage (2026-06-22)

A read-only, multi-agent audit of POE2GPS (perf / security-footprint / code-quality / correctness).
**Headline: invariants are CLEAN** — tree-wide grep confirms zero write/inject/input APIs under `src`,
`OpenProcess` uses the minimal `PROCESS_VM_READ | PROCESS_QUERY_LIMITED_INFORMATION` mask, no
identifying data (character name) in any payload, and the compliance gate is sound. Totals: 4 high,
12 medium, 13 low.

## ✅ Applied autonomously (shipped in v0.1.6)

Safe, verifiable without a live client (build/test/gate/grep):

- **Dropped raw `addr` from `/entities`** (`ApiServer.cs`) — the live in-process entity address was on
  the wire and unused by the dashboard; a tooling-fingerprint field. Removed; entity `id` already keys
  the client.
- **Generic 500 responses** (`ApiServer.cs`) — error bodies no longer echo raw exception messages (which
  can carry memory addresses / struct names); detail is logged locally instead.
- **Update-check opt-out** (`RadarSettings.CheckForUpdates`, default on) — gates the one startup GitHub
  request; turn off in Settings → "Check for updates" for zero network egress.
- **Deleted dead code** — `FogBitmap` + `ExplorationTracker` (PoE1-fork fog-of-war leftovers,
  grep-verified zero references).

## ⏸ Deferred — need a live client to validate (greenlight when you're at the game)

These touch the **memory-read hot path**; a subtle wrong-read is invisible without the game, so I won't
ship them unsupervised. All are pure read-batching/caching — **no offset changes.**

1. **`ResolveComponent` single-pass bulk read** *(highest perf win)* — `Poe2Live.cs:1353-1378`. Today it
   issues one `ReadStringUtf8` syscall **per bucket entry** and runs separately per component per new
   entity, so a freshly-spotted monster pack is the worst-case tick (hundreds of RPM calls). Bulk-read
   the contiguous `ComponentLookUp` bucket once, match the handful of component names in-process, index
   `ComponentList` once → 5 bucket-walks collapse to 1. *Validate via the dashboard Entities view.*
2. **Cache `Reaction` per entity** — `Poe2Live.cs:638-647`. Hostility is read live for every entity every
   tick (~6–12k syscalls/sec) but is static per spawn. Cache it like rarity (evict on `EvictEntity`),
   optional slow-cadence refresh for the rare conversion case. ~25% of the per-entity syscall budget.
3. **`BuildNavTargets` LINQ→foreach + memoize `Resolve`** — `RadarApp.cs:1332-1363`. Per-tick iterator
   chain + ValueTuple-per-entity + a full `OrderBy` that's dead work (`AddNearestPathTarget` does its own
   nearest-scan); `_displayRules.Resolve(e)` runs 2–3× per entity per tick. Single-pass + drop the sort +
   memoize once per tick. (CPU/alloc only — not a memory-read change, so lower-risk; deferred mainly to
   keep nav behavior validated.)
4. **Throttle `DirectorReconcile`** to 2–4 Hz — `RadarApp.cs:1419-1433`. Ranks the full set every tick;
   the decision changes rarely. (Experimental feature; low risk.)
5. **Atlas signature caching + render-thread alloc trims** — cache the per-entity census signature; return
   the atlas projection via a struct instead of a fresh `double[8]` each frame; gate route-list rebuilds
   on the existing `_lastAtlasSig`.

### Correctness hardening (deferred — read-path/threading)
- **Vital-offset auto-heal never re-validates** (`Poe2Live.cs:244-285`) — a one-shot latch; a bad first
  anchor sticks for the session, and the three readers can latch different offsets. Add an opportunistic
  re-heal when a latched read looks wrong.
- **std::map walk cap should scale with `size`** (`Poe2Live.cs:335-348`) — flat 200k cap can balloon to
  ~200k syscalls on a torn tree during zone load (multi-hundred-ms world-thread stall). Bound to
  `size*4 + 1024`.
- **Map-UI / HP-bar specs lack a Self-pointer liveness check** (`Poe2Live.cs:938-965`, `1257-1263`) — a
  recycled slot can project a phantom marker for <33 ms. Add the same `Self == el` guard used elsewhere.
- **`WorldLoop` should `PublishEmptyWorld()` on a thrown tick + rate-limit the error log**
  (`RadarApp.cs:604-622`) — a persistently-throwing tick spams stderr 30×/s and leaves stale data.

## 🏗 Code-structure (future dedicated sessions — bigger, no behavior change)

- **Split `RadarApp.cs`** (2187 lines) into `RadarApp` (thread orchestration) + `NavigationController` +
  `AtlasController` + `RewardPanelReader` + a Win32 monitor/key helper. The nav/atlas clusters are
  already lock-isolated (`_navLock`/`_atlasLock`) → safest to extract first, one per commit.
- **Centralize JSON load/save boilerplate** — add `JsonStore.TryLoad<T>`/`SafeWrite<T>`; ~16 duplicated
  blocks across 8 stores collapse to one-liners. (Mechanical; deferred only to avoid an unsupervised
  8-file sweep.)
- **Extract the settings parse/sanitize cluster** from `ApiServer.cs` (~250 lines) into a `SettingsApplier`.
- **Move the dashboard** out of the 1546-line C# raw-string literal into embedded `.html/.css/.js`
  resources (editor ergonomics + JS linting).
- **Consolidate hex-color parsing** (3 forks: `ApiServer`/`OverlayRenderer`/`RadarApp`) into one helper.
